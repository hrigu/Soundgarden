# 28 - Enhancement (Spatial-Audio): Konstruktor-Aufrufe auf Keyword-Args umstellen

## 1. Why

### 1.1 Objective
Mehrargumentige Aufrufe von `sg`-Konstruktoren/Methoden lesbarer machen: Keyword-Args statt langer positionaler Listen, v.a. wo mehrere Zahlen ohne Typunterschied leicht vertauschbar sind (z.B. `InsectSound.new(60, 0.6, 800, 1400, 0.08, 0.06, 0.05)`).

### 1.2 Context
Alle betroffenen `*new`-Methoden deklarieren ihre Parameter bereits mit Namen (teils mit Defaults) — SuperCollider unterstützt Keyword-Args also schon technisch. Es fehlt nur die Umstellung an den Aufrufstellen. `Room.sc`/`demos/insects.scd` nutzen an einer Stelle (`addSynthDef(AtkBinauralizer, subjectID: 21)`) bereits diesen Stil — das wird als Vorbild fortgeführt.

## 2.0 What

```gherkin
Feature: Keyword-Args an Konstruktor-Aufrufen

  Als Person, die den Code liest oder erweitert
  möchte ich an Aufrufstellen mit mehreren Argumenten sehen, welcher Wert zu welchem Parameter gehört
  damit ich nicht in der Methodensignatur nachschlagen muss, um z.B. `InsectSound.new(60, 0.6, 800, 1400, 0.08, 0.06, 0.05)` zu verstehen.

  Scenario: InsectSound-Aufrufe benennen ihre Parameter
    Given demos/insects.scd mit InsectSound.new(280, 0.2, 5200, 7000, 0.015, 0.01, 0.3)
    When ich auf Keyword-Args umstelle
    Then steht z.B. wingRate: 280, amp: 0.3 explizit im Aufruf, Klangverhalten bleibt identisch

  Scenario: Nur nicht-triviale Default-Abweichungen werden benannt
    Given ein Aufruf, der für die meisten Parameter den Default übernimmt
    When ich umstelle
    Then werden nur die tatsächlich abweichenden Parameter als Keyword übergeben, der Rest bleibt weg
```

## 3.0 How

### 3.1 Implementation Context
Reine Aufrufstellen-Änderung, keine Signatur-/Logikänderung. Betroffen:

- `InsectSound.new` — `demos/insects.scd` (2 Aufrufe, 7 Positionsargumente)
- `SoundObject.new(movable, sound, binauralizer)` — `classes/sg/spatial/Room.sc`, `demos/one_insect.scd`
- `Movable.new(pos, moveRule, roomRadius)` — `tests/sg/soundobjects/TestMovable.sc`, `demos/sample.scd`, `demos/sample_moving.scd`, `demos/reverb.scd`, `demos/one_insect.scd`, `demos/insects.scd`
- `CircularMoveRule.new(baseRadius, breathAmount, breathRate, angularSpeed, startAngle)` — `tests/sg/soundobjects/TestCircularMoveRule.sc`, `demos/sample_moving.scd`, `demos/insects.scd`
- `Listener.new(pos, facing, binauralizerClass)` — `tests/sg/spatial/TestListener.sc`
- `SampleSound.new(path, rate, phase)` — `demos/sample.scd`, `demos/sample_moving.scd`, `demos/reverb.scd`
- `Room>>register(movable, sound, reverbMix)` — `demos/reverb.scd`, `tests/sg/spatial/TestRoom.sc`
- `Binauralizer>>play(server, inBus, outBus, target, addAction)`-Aufruf in `classes/sg/soundobjects/SoundObject.sc`

Nur Argumente benennen, die vom Default abweichen bzw. zur Lesbarkeit beitragen; reine `pos`/`path`/Objekt-Argumente (die schon durch ihren Typ erkennbar sind) bleiben positional, wo das nicht verwirrend ist — Einzelfallentscheidung pro Zeile.
