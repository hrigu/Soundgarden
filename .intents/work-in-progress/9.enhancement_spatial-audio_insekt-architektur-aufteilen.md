# 9 - Enhancement (Spatial-Audio): Insekt-Architektur in Sound, Bewegungsregel und Binauralisierung aufteilen

## 1. Why

### 1.1 Objective

Klare Verantwortlichkeiten trennen: Der Insektenklang selbst soll nicht wissen, wo er im Raum ist. Die Bewegungsregel soll ein austauschbares Objekt sein statt einer rohen Function. Und eine eigene Klasse soll dafür zuständig sein, wie der Klang für die Ohren des Hörers geformt wird (Pan, Lautstärke, Klangfarbe) — abhängig von Distanz und Ausrichtung zwischen Listener und Insekt.

### 1.2 Context

Intent 3 hat einen ersten funktionierenden Prototyp geliefert, aber pragmatisch alles in einer SynthDef gebündelt (Klangerzeugung + Pan/ITD/Distanz-Filterung in einem) und die Bewegungsregel als rohe Function übergeben. Der Nutzer möchte jetzt strikt objektorientiert weiterbauen, bevor mit Intent 5 (ATK-HRTF) die Binauralisierung ausgetauscht wird — eine saubere Trennung jetzt erspart, das HRTF-Rendering später aus einer verwobenen SynthDef herausschälen zu müssen.

### 1.3 Depends On Intents

* 3 - Feature (Spatial-Audio): Binaurales Insekt-Prototyp — liefert die Struktur, die hier aufgeteilt wird.

### 1.4 Related to Intents

* 5 - Feature (Spatial-Audio): ATK-basiertes HRTF für das Insekt — wird künftig auf der neuen `Binauralizer`-Klasse aufsetzen statt direkt auf der `insectVoice`-SynthDef.

## 2.0 What

```gherkin
Feature: Insekt-Architektur in eigenständige Klassen aufgeteilt

  Als Entwickler, der die räumliche Klanginstallation weiterbaut
  möchte ich Klang, Bewegungsregel und Binauralisierung als eigenständige Klassen
  damit ich jede davon unabhängig austauschen kann (z.B. später ATK-HRTF statt
  einfachem Pan/ITD), ohne die jeweils anderen Verantwortlichkeiten anzufassen.

  Scenario: Insektenklang kennt seine Position nicht
    Given eine InsectSound-Instanz mit Default-Parametern
    When sie einen Klang erzeugt
    Then enthält ihr Signal keinerlei Pan-, Distanz- oder Azimuth-Bezug

  Scenario: Insektenklang mit angepassten Parametern
    Given eine InsectSound-Instanz mit expliziten Klangparametern (z.B.
      Flügelschlag-Rate, Brummfrequenzen)
    When sie gespielt wird
    Then hört man den Klang mit den übergebenen statt den Default-Werten

  Scenario: Binauralisierung übernimmt Pan und Lautstärke
    Given eine Binauralizer-Instanz und Azimuth/Distanz zwischen Listener und Insekt
    When sich Azimuth oder Distanz ändern
    Then passt die Binauralizer-Instanz Pan, ITD, Lautstärke und Klangfarbe
      entsprechend an — unabhängig vom InsectSound

  Scenario: Bewegungsregel als eigenes Objekt
    Given eine MoveRule-Instanz (z.B. Kreisbahn mit Radius/Atmung/Winkelgeschwindigkeit)
    When ein Movable ihr per step() folgt
    Then verwendet es die Regel über eine einheitliche Schnittstelle statt eine
      rohe Function aufzurufen

  Scenario: Bestehendes Verhalten bleibt hörbar identisch
    Given die Kreisbahn aus insect_demo.scd, neu zusammengesetzt aus InsectSound +
      MoveRule + Binauralizer
    When das Insekt abgespielt wird
    Then klingt es wie vorher (Hörtest) — reines Refactoring, keine Verhaltensänderung
```

## 3.0 How

### 3.1 Implementation Context

* `classes/Movable.sc` — `moveRule`-Var hält künftig ein `MoveRule`-Objekt statt einer Function; `step()` ruft `moveRule.next(pos, time, dt)` statt `moveRule.value(...)` auf.
* Neu `classes/MoveRule.sc` — Basisklasse, kapselt eine Bewegungsregel über eine einheitliche `next(pos, t, dt)`-Schnittstelle.
* Neu `classes/CircularMoveRule.sc` — erste konkrete Regel: Kreisbahn mit atmendem Radius (bisher `~circleRule` in `insect_demo.scd`), Parameter (Radius, Atem-Amplitude/-Rate, Winkelgeschwindigkeit) mit sinnvollen Defaults, aber änderbar.
* Neu `classes/InsectSound.sc` — kapselt die reine Klangerzeugung (bisheriger Flügel-/Brumm-Teil aus der `\insectVoice`-SynthDef), spielt auf einen privaten Audio-Bus statt direkt auf den Hardware-Output; Klangparameter (Flügelschlag-Rate, Brummfrequenzen, Resonanz, Basis-Amp) mit Defaults, per `*new`-Argumenten überschreibbar.
* Neu `classes/Binauralizer.sc` — übernimmt Pan, ITD, distanzabhängigen Pegel/Tiefpass, "hinten"-Dämpfung; liest vom `InsectSound`-Bus, schreibt Stereo auf den Hardware-Output. Parameter (ITD-Skalierung, Cutoff-Bereich, Amp-Rolloff, Behind-Damping-Kurve) mit Defaults. Nimmt pro Tick Azimuth/Distanz entgegen (von `Listener` berechnet) — bleibt austauschbar für die spätere ATK-Umsetzung (Intent 5).
* `classes/SoundInsect.sc` — wird zum Orchestrator: hält `Movable` (mit `MoveRule`), `InsectSound`, `Binauralizer`; die bestehende `Routine` tickt Bewegung, berechnet Azimuth/Distanz über `Listener` und schiebt sie in den `Binauralizer` statt in eine monolithische SynthDef.
* `classes/Listener.sc` — unverändert (bleibt zuständig für reine Geometrie: Azimuth-/Distanzberechnung).
* `insect_demo.scd` — inline-SynthDef entfällt; stattdessen `CircularMoveRule` + `InsectSound` + `Binauralizer`-Instanzen zusammenstecken.
* Tests: `MoveRule`/`CircularMoveRule` sind reine sclang-Logik ohne Server-Abhängigkeit → testbar per `UnitTest` (analog `TestBootTrackDetection`). `InsectSound`/`Binauralizer` sind SynthDef-lastig → Verifikation per Hörtest, kein TDD-Zwang (siehe AGENTS.md TDD-Abschnitt).
