# 55 - Enhancement (Spatial-Audio): SpatialControlPanel aufteilen

## 1. Why

### 1.1 Objective

`SpatialControlPanel.sc` (629 Zeilen) in kleinere, einzeln verständliche und (wo sinnvoll)
einzeln testbare Bausteine aufteilen, statt einer God Class, die Fenster-Lifecycle,
Maus-/Tastatur-Input, Welt↔Bildschirm-Geometrie, Zeichnen, Solo/Mute-Logik und
Preset-/Slider-GUI-Aufbau in einer Klasse vermischt.

### 1.2 Context

Architektur-Review (siehe vorheriges Gespräch) identifizierte `SpatialControlPanel` als
einzige Stelle im Projekt, die spürbar über "eine Klasse, eine Aufgabe" hinausgeht.
`tests/sg/gui/TestSpatialControlPanel.sc` instanziiert die Klasse bereits direkt mit Fakes
(kein echtes GUI) für Solo-Logik und `sampleSelectionRangeFor` — diese Tests müssen mit den
entsprechenden Verantwortlichkeiten mitwandern, nicht nur der Produktivcode.

## 2.0 What

```gherkin
Feature: SpatialControlPanel aufteilen

  Als Entwickler an diesem Repo
  möchte ich SpatialControlPanel nicht mehr als eine 629-Zeilen-God-Class vorfinden
  damit Verantwortlichkeiten einzeln verständlich, testbar und änderbar bleiben.

  Scenario: Solo/Mute-Logik unabhängig testbar
    Given SoloMuteController als reine Logik-Klasse ohne GUI-Bezug
    When die bestehenden Solo-Szenarien (Restore-vor-Stummschaltung, Disable-Solo,
      Solo-ohne-Auswahl) gegen SoloMuteController laufen
    Then verhalten sie sich identisch zum bisherigen SpatialControlPanel-Verhalten

  Scenario: Sample-Preview-Logik unabhängig testbar
    Given SoundObjectControlsView als eigene Klasse
    When die bestehenden sampleSelectionRangeFor-Szenarien dagegen laufen
    Then verhalten sie sich identisch zum bisherigen SpatialControlPanel-Verhalten

  Scenario: Demos unverändert nutzbar
    Given ein Demo-Skript, das RoomControlPanel/SpatialControlPanel wie bisher verwendet
    When es nach der Aufteilung ausgeführt wird
    Then verhält sich das GUI (Draufsicht, Regler, Solo, Presets) unverändert
```

## 3.0 How

### 3.1 Implementation Context

Reine Struktur-Änderung, kein beabsichtigter Verhaltensunterschied. Neue Klassen liegen wie
bisher unter `classes/sg/gui/`, Tests gespiegelt unter `tests/sg/gui/`. Natürliche Nähte
(bereits durch Kommentar-Abschnitte im bestehenden Code sichtbar):

- **`SpaceCanvas`** — linke Draufsicht komplett: Zeichnen, Welt↔Bildschirm-Transform,
  Hit-Testing, Maus-Dragging von Listener/Soundobjekt, Tastatursteuerung (W/S/A/D/Q/E) des
  Listeners. Meldet Auswahländerungen per Callback.
- **`SoloMuteController`** — reine Logik-Klasse (kein GUI): bisherige
  Solo/Mute-Zustandsmaschine (`soloSelectedOnly`/`soloAmpCache`/Restore-Invariante).
- **`SoundObjectControlsView`** — dynamischer rechter Bereich fürs ausgewählte Soundobjekt
  (Titel, Solo-Button, Sample-Waveform-Preview inkl. `sampleSelectionRangeFor`,
  Parameter-Slider, Preset-Laden/Speichern).
- **`RoomParamsView`** — statischer oberer Bereich (Room-/Hall-Regler + ReverbSend-Regler).
- **`SpatialControlPanel`** — bleibt schlanker Orchestrator: Fenster/Lifecycle, komponiert
  die vier obigen Bausteine, verdrahtet Auswahländerung zwischen `SpaceCanvas` und den beiden
  rechten Panels.
