# 39 - Enhancement (Spatial-Audio): Steuerung, Raum-Draufsicht und Room-GUI in gemeinsames GUI vereinen

## 1. Why

### 1.1 Objective
Die bisher getrennten Fenster für Listener-Steuerung, editierbare Raum-Draufsicht und
Room-/Reverb-Regler in ein gemeinsames GUI zusammenführen, das in allen Spatial-Audio-Demos
wiederverwendet werden kann.

### 1.2 Context
Aktuell gibt es mehrere getrennte GUI-Bausteine:

* `KeyboardListenerControl` — eigenes fokussiertes Fenster nur für W/S/A/D/Q/E
* `SpaceView` — eigenes Fenster für die 2D-Draufsicht, inzwischen optional mit
  Drag-and-drop der Klangquellen
* `reverb_analysis.scd` — eigenes Reverb-Fenster für `size`, `height`, `surface`, `mix`,
  `reverbMix`

Für den praktischen Demo-Einsatz ist das unnötig zerstreut:

* mehrere Fenster statt eines Arbeitsbereichs
* Fokus muss zwischen Fensterrollen mitgedacht werden
* neue GUI-Funktionen werden nur im Demo verdrahtet statt als allgemeine Infrastruktur

Ein gemeinsames GUI macht den Workflow kohärenter: Raum sehen, Listener bewegen, Klangquellen
ziehen und Hall-Parameter ändern — alles in einer Oberfläche.

### 1.3 Related To Intents

* 16 - Beweglicher Listener per Tastatur
* 19 - Space View
* 35 - Raumhall-Analyse-Demo
* 38 - RoomReverb stop/play restart-sicher machen

## 2.0 What

```gherkin
Feature: Gemeinsames Spatial-Control-GUI

  Scenario: Listener-Steuerung und Raumübersicht in einem Fenster
    Given ein Orchestra mit Listener und Soundobjekten
    When ich das gemeinsame GUI öffne
    Then sehe ich die Raum-Draufsicht und kann im selben Fenster den Listener per Tastatur steuern

  Scenario: Klangquellen in derselben Oberfläche verschieben
    Given das GUI ist im editierbaren Modus geöffnet
    When ich ein Soundobjekt in der Draufsicht mit der Maus ziehe
    Then wird seine Position live geändert

  Scenario: Room-Parameter im selben GUI ändern
    Given das gemeinsame GUI ist geöffnet
    When ich size, height, surface, mix oder reverbMix ändere
    Then aktualisiert sich der laufende Room sofort
    And ich muss dafür kein separates Reverb-Fenster öffnen

  Scenario: Wiederverwendung in mehreren Demos
    Given ein Demo nutzt bisher KeyboardListenerControl und SpaceView getrennt
    When es auf das gemeinsame GUI umgestellt wird
    Then braucht es nur noch einen gemeinsamen Einstiegspunkt statt mehrerer Fenster
```

## 3.0 How

### 3.1 Implementation Context
Neue gemeinsame Klasse `classes/sg/spatial/SpatialControlPanel.sc`, statt
`KeyboardListenerControl` und `SpaceView` hart zu verschmelzen.

Die Klasse kapselt:

* ein einzelnes `Window`
* eine linke `UserView` für die Draufsicht
* einen rechten Bedienbereich für Room-/Reverb-Regler
* Tastatur-Handling für den Listener
* optional Drag-and-drop der Soundobjekte
* Room-/Reverb-Regler (`size`, `height`, `surface`, `mix`, `reverbMix`)
* gleiches `play`/`stop`-Lifecycle-Muster wie die bestehenden GUI-Klassen

Geplante API:

* `*new { |room, viewRadius = 8, moveSpeed = 2, rotateSpeed = 90, editable = true| ... }`
  — nimmt bewusst `room`, nicht nur `orchestra`, damit das Panel sowohl die Draufsicht
  (`room.orchestra`) als auch die Room-Regler (`room.size = ...`) direkt bedienen kann.
* `play { |updateRate = 20| ... }` — öffnet genau ein Fenster, aktiviert Tastatursteuerung,
  Refresh-Routine und Regler
* `stop` — beendet Routine und schließt das Fenster
* interne Hilfsmethoden für
  `draw`
  `soundObjectAtScreenPoint`
  `moveSoundObjectToScreenPoint`
  `applyReverbMix`
  `worldToScreen` / `screenToWorld`

Bestehende Klassen bleiben entweder:

* vorerst bestehen und werden in den Demos schrittweise ersetzt, oder
* werden später auf gemeinsame Hilfsmethoden reduziert; das ist nicht Teil dieses Intents

Zuerst umstellen:

* `demos/reverb/reverb_analysis.scd`
* danach bei Bedarf weitere Demos (`sample.scd`, `sample_moving.scd`, `grill/insects.scd`,
  `grill/three_insects.scd`)
