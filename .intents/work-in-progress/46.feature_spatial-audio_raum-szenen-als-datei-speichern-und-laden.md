# 46 - Feature (Spatial-Audio): Raum-Szenen als Datei speichern und laden

## 1. Why

### 1.1 Objective
Den aktuellen Zustand einer Spatial-Audio-Szene als Datei speichern und später wieder laden
können, damit Klangkonstellationen, Raumparameter und Objekt-Settings für Kompositionen
reproduzierbar bleiben — sowohl über die GUI (bestehende Szene ersetzen) als auch direkt im
Code (Szene als initiale Konfiguration eines Demo-Skripts laden).

### 1.2 Context
Mit Intent 43 gibt es bereits Presets für einzelne Soundobjekte (aktuell nur Sound-Parameter,
keine Position/Bewegung). Für die weitere kompositorische Arbeit reicht das nicht aus, weil die
musikalisch relevante Einheit oft die gesamte Szene ist: Raum, Listener, Objektpositionen,
Bewegungsregeln und mehrere Klangobjekte mit ihren individuellen Parametern. Ohne
Szenen-Speicherung müsste dieser Zustand jedes Mal manuell rekonstruiert werden.

Gewünschtes Zielbild:

* im Demo eine interessante Konstellation bauen
* diesen Gesamtzustand als Datei sichern
* ihn später wieder laden und genau an diesem Stand weiterarbeiten — entweder als initiale
  Konfiguration eines neuen Demo-Laufs, oder um eine bereits laufende Szene per GUI zu ersetzen

### 1.3 Related To Intents

* 43 - Live-editierbare Grillen-Sounds (synthetisch + Sample) mit Preset-Bibliothek
* 44 - Einfaches RoomControlPanel als Demo-Wrapper
* 45 - Preset-Bereich im Control-Panel vollständig bedienbar machen
* 55 - SpatialControlPanel aufteilen (RoomParamsView, wo die neue Szenen-UI andockt)

## 2.0 What

```gherkin
Feature: Gesamte Spatial-Audio-Szene speichern und laden

  Scenario: Aktuelle Szene speichern
    Given ein Room mit Listener und mehreren Soundobjekten
    When ich die Szene unter einem Namen speichere
    Then wird eine Datei mit allen relevanten Raum-, Listener- und Objekt-Parametern angelegt

  Scenario: Gespeicherte Szene per GUI in eine laufende Szene laden
    Given eine zuvor gespeicherte Szene
    When ich sie im Control-Panel lade
    Then werden die aktuellen Soundobjekte gestoppt und Room/Listener/Soundobjekte mit den
      gespeicherten Werten neu aufgebaut

  Scenario: Gespeicherte Szene als initiale Konfiguration im Code laden
    Given eine zuvor gespeicherte Szene und ein gebooteter Server
    When ein Demo-Skript RoomSceneLibrary.build mit Server und Listener aufruft
    Then werden die benötigten SynthDefs registriert und ein fertig aufgebauter Room
      zurückgegeben, den das Skript direkt abspielen kann

  Scenario: Ein Preset für ein Soundobjekt umfasst auch Position und Bewegung
    Given ich habe ein Soundobjekt im Raum verschoben und seine Klangparameter angepasst
    When ich sein Preset speichere und später auf ein anderes Objekt anwende
    Then übernimmt das andere Objekt Klang, Position und Bewegungsregel des Presets

  Scenario: Unterschiedliche Sound-Klassen werden korrekt serialisiert
    Given die Szene enthält z.B. CricketSound, SampleSound oder SongSound
    When ich speichere und wieder lade
    Then bleiben Sound-Klasse, ihre editierbaren Parameter und (bei SampleSound/SongSound)
      der Datei-Pfad erhalten

  Scenario: Alte, positionslose Objekt-Presets bleiben ladbar
    Given ein vor Intent 46 gespeichertes Preset ohne movable-Daten
    When ich es lade und anwende
    Then werden Sound-Parameter übernommen und die Position unverändert gelassen (kein Error)
```

## 3.0 How

### 3.1 Implementation Context

* `MoveRule` bekommt `*editableParams` (Basisklasse `[]`) + `setParam` (analog zu
  `Sound>>setParam`, ohne Synth-Bezug); `CircularMoveRule` deklariert seine 5 Parameter
  (baseRadius, breathAmount, breathRate, angularSpeed, startAngle). `SteadyMoveRule` erbt die
  leere Liste unverändert.
* `Sound` bekommt `*requiredConstructorArgs` (Basisklasse `[]`) + generisches
  `*buildFromSavedParams { |params| ... }`: baut eine Instanz aus den Pflicht-Konstruktor-
  Argumenten (in Deklarationsreihenfolge) und wendet danach `editableParams` per `setParam`
  an. `SampleSound`/`SongSound` überschreiben `requiredConstructorArgs` mit `[\path]`.
* `SoundPresetLibrary` → umbenannt zu `SoundObjectPresetLibrary`, verschoben nach
  `classes/sg/soundobjects/` (Domäne jetzt SoundObject statt reiner Sound). Preset-Format
  erweitert um ein verschachteltes `movable:`-Event (moveRuleClass, pos, roomRadius,
  MoveRule-`editableParams`). `*save`/`*applyTo` nehmen jetzt ein `SoundObject` statt eines
  bloßen `Sound` entgegen. Alte Presets ohne `movable`-Schlüssel bleiben ladbar — beim Anwenden
  wird die Positions-/Bewegungs-Wiederherstellung dann einfach übersprungen.
* Neue `RoomSceneLibrary` (`classes/sg/spatial/`):
  * `*save { |dir, name, room| ... }` — Room-Parameter (size/height/surface/mix/spread/
    inputBandwidth/tailBalance), Listener (pos/facing) und alle Soundobjekte (über
    `SoundObjectPresetLibrary`s Serialisierungs-Helper) in eine Datei.
  * `*load { |dir, name| ... }` — liefert das rohe Event (wie `SoundObjectPresetLibrary.load`).
  * `*applyTo { |room, sceneEvent| ... }` — für die GUI: stoppt/entfernt alle aktuellen
    Soundobjekte des Room (`Orchestra>>clear`, neu), setzt Room-/Listener-Parameter, baut alle
    Soundobjekte aus dem Event neu auf und registriert sie.
  * `*build { |dir, name, server, listener| ... }` — für den Code-Einstieg: lädt die Datei,
    ruft `addSynthDef` auf jeder in der Szene vorkommenden Sound-Klasse auf (Klassenname steht
    ohnehin pro Soundobjekt in der Datei, keine separate SynthDef-Liste nötig), konstruiert
    einen frischen `Room` mit den gespeicherten Parametern und dem übergebenen Listener, baut
    alle Soundobjekte neu auf und registriert sie — gibt den fertigen `Room` zurück. Nutzt
    intern denselben Soundobjekt-Wiederaufbau wie `applyTo`. Gleiches Zwei-Block-Muster wie
    bestehende Demos (Setup-Block mit `addSynthDef`+`build`, danach separater `~room.play`-
    Block) gibt den asynchronen SynthDef-Registrierungen genug zeitlichen Abstand, ohne
    zusätzlichen Sync-Mechanismus.
  * Bewusste Einschränkung: `RoomSceneLibrary` registriert nur die SynthDefs der in der Szene
    vorkommenden Sound-Klassen — Binauralizer-/Reverb-SynthDefs (schon durch `Room.new`/
    `Listener>>setup` abgedeckt) sind nicht Teil dieser Zuständigkeit.
* GUI: neuer Speichern/Laden-Bereich oben in `RoomParamsView` (Namensfeld + zwei Buttons,
  analog zur bestehenden Objekt-Preset-UI in `SoundObjectControlsView`), dafür ein neuer
  `scenesDir`-Parameter durchgereicht (analog zu `presetsDir`) durch `SpatialControlPanel`/
  `RoomControlPanel`. Aktiviert in den Demos, die schon `presetsDir` nutzen: `crickets.scd`,
  `crickets_with_breaks.scd`, `sample.scd`.
* Bewusst ausser Scope: Laufzeit-Zustände wie das exakte `CallingPattern` einer Grille werden
  beim Laden nicht exakt reproduziert, sondern beim Konstruieren frisch neu gewürfelt (z.B.
  `CricketSound.new` ohne `pattern`-Override erzeugt automatisch ein neues) — nur Sound-Timbre,
  Position und Bewegungsregel gelten als "der Zustand" einer Szene.
