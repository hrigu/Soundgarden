# 20 - Feature (Spatial-Audio): Rhythmische Sample-Wiedergabe im Kreis

## 1. Why

### 1.1 Objective

Klangobjekte, die statt eines synthetisierten Klangs (wie `InsectSound`) ein Audiosample in
einem festen Rhythmus abspielen, als eigenständiges Experiment: mehrere steady Soundobjekte
im Kreis um den Listener angeordnet.

### 1.2 Context

Bisher gibt es nur synthetisierte Klänge (`InsectSound`). Für perkussive/rhythmische Texturen
sind echte Samples (z.B. die vorhandenen `sounds/*.aiff`-Testdateien) interessanter. Als
Explorations-Schritt zunächst ein eigenständiges `sample_demo.scd`, keine Integration in
`insect_demo.scd` — spätere Zusammenführung beider Prototypen ist eine mögliche
Weiterentwicklung, aber nicht Teil dieses Intents.

### 1.3 Depends On Intents

* 11 - Enhancement (Spatial-Audio): Sound-Superklasse einführen — liefert die Basisklasse
  (`Sound`), die hier um `SampleSound` erweitert wird.
* 15 - Feature (Spatial-Audio): Steady Soundobjekte — liefert `SteadyMoveRule`/`moveTo`, mit
  denen die Objekte im Kreis platziert werden.

## 2.0 What

```gherkin
Feature: Rhythmische Sample-Wiedergabe im Kreis

  Als Hörer mit Kopfhörern
  möchte ich mehrere Klangobjekte im Kreis um mich herum hören, die je ein Sample rhythmisch
  abspielen
  damit ich räumliche Rhythmus-Texturen mit echten Samples statt Synthese ausprobieren kann.

  Scenario: Sample wird rhythmisch getriggert
    Given ein SampleSound mit einem geladenen Audiosample und einer Trigger-Rate
    When der Synth läuft
    Then startet das Sample in der gegebenen Rate immer wieder von vorne (kein Loop, kein
      Überlappen)

  Scenario: Mehrere Sample-Objekte im Kreis
    Given N SoundObjects mit SteadyMoveRule, gleichmässig auf einem Kreis um den Ursprung verteilt
    When sample_demo.scd Block 2 ausgeführt wird
    Then hört man über Kopfhörer jedes Objekt an seiner Position auf dem Kreis, Listener bleibt
      im Zentrum

  Scenario: Eigenständiges Experiment
    Given sample_demo.scd
    When es unabhängig von insect_demo.scd ausgeführt wird
    Then entstehen keine Seiteneffekte auf/Abhängigkeiten von insect_demo.scd oder dessen Klassen
      über die ohnehin gemeinsame Spatial-Audio-Infrastruktur hinaus (Listener/Orchestra/
      SoundObject/Movable/Binauralizer)
```

## 3.0 How

### 3.1 Implementation Context

* Neue Klasse `classes/sg/sounds/SampleSound.sc` (`Sound`-Subklasse, analog `InsectSound`):
  lädt einen Buffer (`Buffer.read(server, path)`) in `makeSynth`, SynthDef triggert
  `PlayBuf.ar` per `Impulse.kr(rate, phase)` (retriggert `PlayBuf` per `trigger`-Arg immer neu
  von vorne — kein Loop). `stop` überschrieben, um zusätzlich den Buffer freizugeben.
* Keine Änderung an `SoundObject`/`Orchestra`/`Movable`/`Listener`/`Binauralizer` nötig — volle
  Wiederverwendung, `SampleSound` ist nur ein weiterer `Sound`. `SteadyMoveRule` + `moveTo`
  (oder direkt Startposition) platziert jedes Objekt auf dem Kreis (gleiche
  `[r*sin(angle), r*cos(angle), 0]`-Formel wie in `CircularMoveRule`).
* Neues `sample_demo.scd` (Vorbild `insect_demo.scd`): lädt N Samples aus `sounds/`, verteilt
  N SoundObjects gleichmässig auf einem Kreis, Listener bleibt bei `[0,0,0]` (keine
  Tastatursteuerung nötig für den ersten Wurf). Binauralizer-Variante: `AtkBinauralizer`
  (bereits eingerichtet, besseres Vorne/Hinten — gerade bei mehreren Quellen im Kreis
  relevant).
* Bewusst kein Loop/Overlap-Handling für schnelle Rates > Sample-Länge in diesem Intent
  (einfachste Version zuerst) — als Notiz für später, falls hörbar störend.

## Challenges & Solutions

Beim ersten Testlauf: `WARNING: keyword arg 'channels' not found in call to Meta_Buffer:read`
gefolgt von `Buffer UGen: no buffer data` für jedes Sample-Objekt. Ursache: `channels` ist kein
Argument von `Buffer.read` — das gibt es nur bei der separaten Methode `Buffer.readChannel`
(liest gezielt einzelne Kanäle einer Mehrkanal-Datei). `Buffer.read(..., channels: [0])` lud
dadurch effektiv ohne Kanalauswahl, offenbar mit einem ungültigen/leeren Buffer als Resultat.
Fix: `Buffer.readChannel(server, path, channels: [0])` statt `Buffer.read`.

Rhythmus und Räumlichkeit danach bestätigt hörbar gut — aber `Buffer UGen: no buffer data`
erscheint weiterhin einmal pro Sample-Objekt beim Start. Ursache diesmal harmlos, aber real:
`Buffer.readChannel` lädt die Datei asynchron; `orchestra.play(s)` (Block 2) erzeugt die
Synths aber sofort im Anschluss, `makeSynth` lud den Buffer bis dahin noch gar nicht (das
passierte erst dort, nicht vorher) — der allererste UGen-Check trifft dadurch kurz auf einen
leeren Buffer, bevor die Datei fertig geladen ist (Klang selbst bereits richtig, da der erste
tatsächliche Trigger etwas später kommt). Fix: `SampleSound>>preload(server)` ergänzt, das den
Buffer schon vor `play()` lädt; `sample_demo.scd` ruft das direkt nach dem Anlegen jedes
`SampleSound` in Block 1 auf — gleiches "echte Zeit zwischen den Blöcken"-Muster wie bei den
SynthDef-Registrierungen.
