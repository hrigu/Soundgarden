# 29 - Feature (Spatial-Audio): SoundObject aus Event-Konfiguration bauen, klonbar per Event-Merge

## 1. Why

### 1.1 Objective
SoundObjects aus reinen `Event`-Konfigurationen erzeugen, statt `Movable`/`Sound` von Hand zu
instanzieren und an `SoundObject.new` zu übergeben. Varianten entstehen durch natives
Event-Merge (`++`): von einer bestehenden Konfiguration ausgehend nur einzelne Parameter
überschreiben, um Varianten zu erzeugen (z.B. mehrere Insekten mit ähnlichem Klang, aber
unterschiedlicher Lautstärke/Frequenz) — ohne eigene Builder-Klasse mit Zustand, Klon- oder
Merge-Methoden, da `Event` das (`++`, `.copy`) bereits als SC-Core-Verhalten mitbringt.

### 1.2 Context
`demos/insects.scd` baut aktuell mehrere Insekten von Hand, jedes mit eigenem `Movable`+
`InsectSound`-Verkabelungsblock — Wiederholung ohne Wiederverwendung der gemeinsamen Basis-
Konfiguration. Intent 28 macht die reinen Konstruktor-Aufrufe lesbarer, löst aber nicht das
eigentliche Problem: eine Basis-Konfiguration definieren und daraus mehrere Varianten ableiten.

`Event>>++` liefert bereits ein neues, unabhängiges Event mit vom rechten Operanden
überschriebenen Keys, `Event>>copy` eine unabhängige Kopie — exakt die Merge-/Klon-Semantik,
die ein eigener Builder sonst nachbauen müsste. Der eigentliche Rest-Aufwand ist nur die
Instanziierung: aus einem Params-Event per Setter (`perform`) eine `Movable`/`Sound`-Instanz
bauen und daraus ein `SoundObject`.

`Room>>register` baut den Binauralizer aktuell selbst über `listener.makeBinauralizer(reverbMix)`
und erst dann das `SoundObject` (Intent 27: der Binauralizer-Typ ist eine Eigenschaft der Ohren
des Hörers, nicht der Klangquelle — Skripte kennen ihn nicht). Die Build-Funktion bleibt deshalb
auf movable+sound beschränkt, Room bleibt die einzige Stelle, die den Binauralizer bestimmt.

### 1.3 Related To Intents

* 27 - Binauralizer als Interna des Listeners — Binauralizer-Auswahl bleibt Room/Listener
  vorbehalten, davon weicht dieser Intent bewusst nicht ab.
* 28 - Konstruktor-Aufrufe auf Keyword-Args — macht die rohen Aufrufe lesbarer, dieser Intent
  löst das Wiederholungsproblem bei mehreren Varianten.

## 2.0 What

```gherkin
Feature: SoundObject aus Event-Konfiguration bauen, klonbar per Event-Merge

  Als Person, die mehrere ähnliche Soundobjekte aufbaut
  möchte ich eine Basis-Konfiguration einmal als Event definieren und per Merge (++) variieren
  damit ich nicht bei jeder Variante alle Parameter erneut angeben muss.

  Scenario: SoundObject aus Params-Events erzeugen
    Given ein soundParams-Event (wingRate: 280, amp: 0.3) und ein movableParams-Event
      (moveRule: CircularMoveRule.new)
    When ich SoundObjectBuilder.build(soundParams, movableParams, binauralizer) aufrufe
    Then erhalte ich ein fertiges SoundObject mit einer InsectSound-Instanz (wingRate: 280,
      amp: 0.3, alle anderen Parameter Default), einer Movable-Instanz mit der gegebenen
      MoveRule und dem übergebenen Binauralizer

  Scenario: Variante durch Event-Merge statt erneuter Voll-Konfiguration
    Given ein Basis-soundParams-Event (wingRate: 280, amp: 0.3)
    When ich die Variante baseSoundParams ++ (amp: 0.05) bilde und daraus baue
    Then hat die Variante wingRate weiterhin 280 (übernommen von der Basis) und amp neu 0.05
    And das ursprüngliche Basis-Event bleibt unverändert (Event>>++ liefert ein neues Event)

  Scenario: Zwei aus denselben Params-Events gebaute SoundObjects sind unabhängig
    Given dieselben Params-Events, zweimal durch SoundObjectBuilder.build in ein SoundObject
      umgewandelt
    When ich die Position des einen Movable verändere (step/moveTo)
    Then bleibt die Position des anderen SoundObject unberührt (kein geteilter Movable/Sound-
      State)

  Scenario: Room bestimmt weiterhin den Binauralizer
    Given ein Room mit registrierter Binauralizer-Klasse (addSynthDef)
    When ich room.registerFromBuilder(soundParams, movableParams, reverbMix) aufrufe
    Then baut Room den Binauralizer selbst (wie bei register) und reicht ihn an
      SoundObjectBuilder.build weiter — die Build-Funktion selbst kennt keine
      Binauralizer-Klasse
```

## 3.0 How

### 3.1 Implementation Context

* Neue Klasse `classes/sg/soundobjects/SoundObjectBuilder.sc`, **rein Klassenmethoden, kein
  Zustand/keine Instanz** — Merge/Klon übernimmt bereits `Event` (`++`/`.copy`), dafür ist keine
  eigene Logik nötig:
  - `*build { |soundParams, movableParams, binauralizer, soundClass = InsectSound,
    movableClass = Movable| ... }` — pro Params-Event: `instance = aClass.new;
    params.keysValuesDo { |k, v| instance.perform((k ++ "_").asSymbol, v) }`; danach
    `SoundObject.new(movableInstance, soundInstance, binauralizer)`. `binauralizer` optional
    (nil ⇒ `SoundObject`s bestehendes Fallback-Verhalten greift). Setzt voraus, dass die
    Zielklasse für jedes verwendete Param einen `<>`-Setter hat (bei `Movable`/`InsectSound`/
    `SampleSound` bereits durchgehend der Fall). Kein `doesNotUnderstand`-Trick — bewusst die
    schon vorhandenen `<>`-Setter nutzen, keine neue Magie.
  - Kein Binauralizer-Parameter in den Params-Events (siehe 1.2/1.3) — Binauralizer-Auswahl
    bleibt bei Room/Listener, `build` nimmt ihn nur als fertige Instanz entgegen.
  - Varianten entstehen beim Aufrufer per `baseParams ++ (amp: 0.05)` — keine
    projekteigene Merge-/Klon-API, das ist SC-Core-Verhalten von `Event`.
* `classes/sg/spatial/Room.sc`:
  - Neue Methode `registerFromBuilder { |soundParams, movableParams, reverbMix = 0.3,
    soundClass = InsectSound, movableClass = Movable| ... }` — baut den Binauralizer wie
    `register` über `listener.makeBinauralizer(reverbMix)`, ruft
    `SoundObjectBuilder.build(soundParams, movableParams, binauralizer, soundClass,
    movableClass)` statt `SoundObject.new(movable, sound, binauralizer)`.
  - Gemeinsame Logik (`orchestra.register(soundObject); ^soundObject`) aus `register` in eine
    private Hilfsmethode (z.B. `finishRegistering { |soundObject| ... }`) extrahieren, von
    beiden Methoden genutzt. `register` selbst bleibt unverändert in seiner Signatur — bestehende
    Aufrufstellen (`demos/sample.scd`, `demos/sample_moving.scd`, `demos/reverb.scd`,
    `tests/sg/spatial/TestRoom.sc`), die movable+sound direkt bauen, bleiben unangetastet.
* `tests/sg/soundobjects/TestSoundObjectBuilder.sc` — reine sclang-Logik, kein Server nötig:
  testet `build`-Ergebnis-Typen/-Werte (inkl. Default-Klassen und Override per `soundClass`/
  `movableClass`) und dass zwei `build`-Aufrufe mit denselben Params-Events unabhängige
  Instanzen liefern (Event-Merge/-Copy selbst wird nicht erneut getestet — SC-Core-Verhalten).
* `tests/sg/spatial/TestRoom.sc` — Test für `registerFromBuilder` (Fake-Binauralizer-Erzeugung
  wie bestehend, Params-Events liefern Fake-Movable/-Sound-Klassen).
* `demos/insects.scd` — mind. eine Insekt-Variante (Basis-Event + `++`-Override) auf
  `SoundObjectBuilder.build` + `~room.registerFromBuilder` umstellen, um den eigentlichen
  Anwendungsfall zu zeigen.
