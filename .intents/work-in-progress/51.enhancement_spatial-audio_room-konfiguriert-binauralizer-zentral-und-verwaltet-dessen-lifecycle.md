# 51 - Enhancement (Spatial-Audio): Room konfiguriert Binauralizer zentral und verwaltet dessen Lifecycle

## 1. Why

### 1.1 Objective
`Room` so erweitern, dass Demo-Skripte den Binauralizer-Typ nur noch am `Room` konfigurieren
müssen, z.B.:

```sc
room.binauralizer = AtkBinauralizer;
room.binauralizer = Binauralizer;
room.binauralizer = nil;
```

Der `Room` soll danach selbst verantwortlich sein für:

* Laden der passenden SynthDef bzw. ATK-Initialisierung
* Erzeugen passender SoundObject-Binauralizer bei `register`
* sauberes `teardown` beim Session-Ende

### 1.2 Context
Aktuell muss ein Demo-Skript an mehreren Stellen denselben Entscheid verdrahten:

* `room.addSynthDef(...)`
* implizit `listener.binauralizerClass`
* am Ende `AtkBinauralizer.teardown`

Das ist unnötig verteilt. Bei Experimenten wie `sample.scd` wird der Wechsel von
`AtkBinauralizer` auf `Binauralizer` dadurch fehleranfällig und umständlich.

### 1.3 Related To Intents

* 26 - Room mit physikalischen Eigenschaften steuert Hall
* 37 - Room optional ohne Hall konfigurierbar
* 50 - Anzahl der Sample-Objekte in `sample.scd` begrenzen

## 2.0 What

```gherkin
Feature: Room verwaltet den Binauralizer zentral

  Scenario: Room mit AtkBinauralizer
    Given ein Room mit binauralizer = AtkBinauralizer
    When ich den Room verwende
    Then initialisiert der Room ATK selbst
    And neue SoundObjects bekommen AtkBinauralizer
    And teardown gibt die ATK-Ressourcen wieder frei

  Scenario: Room mit einfachem Binauralizer
    Given ein Room mit binauralizer = Binauralizer
    When ich den Room verwende
    Then registriert der Room die passende SynthDef selbst
    And neue SoundObjects bekommen Binauralizer
    And kein ATK-teardown ist nötig

  Scenario: Room ohne Binauralizer
    Given ein Room mit binauralizer = nil
    When ich SoundObjects registriere
    Then werden sie ohne binaurale Nachbearbeitung gebaut
    And das Demo-Skript braucht keinen separaten Binauralizer-Setup-Code

  Scenario: Demo-Skript wechselt den Typ an einer Stelle
    Given ein Demo-Skript mit Room
    When ich nur room.binauralizer ändere
    Then ist kein zusätzlicher addSynthDef- oder teardown-Code mehr nötig
```

## 3.0 How

### 3.1 Implementation Context

* `classes/sg/spatial/Room.sc` wird zur zentralen Stelle für Binauralizer-Konfiguration und
  Lifecycle
* `Listener` soll keine Lifecycle-Verantwortung tragen, nur noch passende Instanzen erzeugen
* `nil` bedeutet: kein binaurales Processing; dafür braucht `SoundObject` ein robustes Verhalten
  ohne Binauralizer
* bestehende Demos sollen auf die neue API vereinfacht werden
* Tests in `tests/sg/spatial/TestRoom.sc` und ggf. benachbarten Testklassen sichern das
  Verhalten ab

### 3.2 Tasks

- [x] 1.0 Ziel-API für `Room` festlegen und gegenüber bestehendem `addSynthDef` sauber migrierbar machen
- [x] 2.0 TDD: Tests für Binauralizer-Konfiguration und Lifecycle im `Room` ergänzen
- [x] 3.0 `Room` so erweitern, dass er Setup/Instanziierung/Teardown für `AtkBinauralizer`, `Binauralizer` und `nil` selbst übernimmt
- [ ] 4.0 Betroffene Demos auf die neue einfache `Room`-Konfiguration umstellen
- [ ] 5.0 Tests laufen lassen und Intent abschliessen

## Challenges & Solutions

Entscheidungen aus Task 1.0:

* `Room` bekommt eine echte Property `binauralizerClass`, die `Binauralizer`, `AtkBinauralizer`
  oder `nil` akzeptiert.
* `Room.new(...)` bleibt ohne explizite Angabe kompatibel und startet weiterhin mit
  `Binauralizer` als Default.
* Für ATK-spezifische Zusatzparameter wird zunächst nur `subjectID = 21` als Room-Property
  ergänzt, damit `room.binauralizerClass = AtkBinauralizer` ohne weitere Skript-Stellen
  funktioniert.
* Der bisherige Einstieg `room.addSynthDef(...)` bleibt als Migrations-Shim vorerst bestehen und
  delegiert intern auf die neue Room-Konfiguration.
* `nil` wird intern nicht als echtes Fehlen eines Objekts behandelt, sondern auf einen internen
  No-op-/Direct-Out-Binauralizer abgebildet. So bleiben `SoundObject.play`, `updateSpatial` und
  `stop` unverändert einfach.
* Der rote Testlauf zeigt erwartungsgemäss, dass `Room` die neue API noch nicht hat
  (`binauralizerClass_` fehlt) — damit ist der TDD-Ausgangspunkt für Task 3 hergestellt.

Entscheidungen aus Task 3.0:

* Neue Klasse `DirectOutBinauralizer` (`classes/sg/spatial/DirectOutBinauralizer.sc`) implementiert
  den No-op-Fall: Mono-Eingang unbearbeitet auf Stereo-Out, `set` ist ein No-op. Gleiche
  Schnittstelle (`play`/`set`/`stop`) wie `Binauralizer`/`AtkBinauralizer`.
* `Room>>binauralizerClass_` erkennt AtkBinauralizer-artige Klassen über `respondsTo(\setup)`
  statt `=== AtkBinauralizer` — damit laufen Test-Doubles (siehe `TestRoom`) ohne echte
  Vererbung durch dieselbe Weiche.
* `nil` wird in `binauralizerClass_` auf `directOutBinauralizerClass` (Default
  `DirectOutBinauralizer`) umgeleitet; dessen `addSynthDef` wird nur aufgerufen, wenn die Klasse
  das überhaupt anbietet (`respondsTo(\addSynthDef)`).
* `teardown` delegiert zusätzlich an `binauralizerClass.teardown`, falls vorhanden
  (`respondsTo(\teardown)`) — deckt den AtkBinauralizer-Kernel-Abbau ab, ohne dass `Room`
  hartcodiert weiss, welche Klasse das braucht.
* Der bisherige `addSynthDef(binauralizerClass, subjectID)`-Einstieg bleibt als Migrations-Shim
  bestehen und delegiert vollständig auf `binauralizerClass_`.
* Headless-Testlauf (`sclang` via `/Applications/SuperCollider.app/Contents/MacOS/sclang`)
  komplett grün, alle `TestRoom`-Fälle inklusive der neuen bestehen.
