# 22 - Feature (Spatial-Audio): Geteilter Raumhall mit distanzabhängigem Send

## 1. Why

### 1.1 Objective

Einen gemeinsamen Raumklang (Hall) für den virtuellen Raum einführen: jedes SoundObject
schickt einen Anteil seines trockenen Signals in einen geteilten Hall, wobei der Anteil mit
der Distanz zum Listener wächst — nah kaum Hall, weit deutlich mehr. Kein Hall pro Objekt.

### 1.2 Context

Aktuell hat jedes Objekt nur einen Direktschall-Pfad (Binauralizer/AtkBinauralizer: Pan/HRTF +
distanzabhängiger Pegel/Tiefpass). Ein Raum hat aber nur EIN Nachklangfeld, nicht eines pro
Quelle — akustisch korrekt und günstig ist ein geteilter Hall, in den alle Quellen
unterschiedlich stark einspeisen. Das Verhältnis Direkt-zu-Diffus ist einer der wichtigsten
Distanz-Hinweise fürs Gehör.

### 1.3 Depends On Intents

* 3 - Feature (Spatial-Audio): Binaurales Insekt-Prototyp — liefert `Binauralizer`.
* 5 - Feature (Spatial-Audio): ATK-basiertes HRTF für das Insekt — liefert `AtkBinauralizer`.

## 2.0 What

```gherkin
Feature: Geteilter Raumhall mit distanzabhängigem Send

  Als Hörer mit Kopfhörern
  möchte ich einen gemeinsamen Raumklang hören, zu dem jedes Objekt je nach Entfernung
  unterschiedlich stark beiträgt
  damit sich der virtuelle Raum akustisch glaubwürdiger anfühlt, nicht nur durch Pegel/Pan.

  Scenario: Ein geteilter Hall für alle Objekte
    Given eine RoomReverb-Instanz und mehrere SoundObjects
    When orchestra.play läuft
    Then hört man einen einzigen gemeinsamen Nachklang, nicht einen pro Objekt

  Scenario: Send-Level wächst mit der Distanz
    Given zwei SoundObjects mit unterschiedlicher Distanz zum Listener
    When beide spielen
    Then trägt das weiter entfernte Objekt hörbar mehr zum Hall bei als das nahe
```

## 3.0 How

### 3.1 Implementation Context

* Neue Klasse `classes/sg/spatial/RoomReverb.sc`: legt einen geteilten Mono-Bus an, ein
  `GVerb`-Synth liest davon und schreibt (Stereo) auf den Hardware-Output — läuft permanent,
  `play`/`stop` wie andere Spatial-Klassen. `GVerb` ist Core-UGen, keine sc3-plugins-Frage.
  `drylevel=0` (Direktschall kommt schon über die Binauralizer, hier nur der Nachklang-Anteil).
* `Binauralizer`/`AtkBinauralizer`: `*addSynthDef` bekommt einen neuen Parameter `reverbBus`
  (Bus der `RoomReverb`-Instanz — muss vor `addSynthDef` existieren, gleiches
  Reihenfolge-Muster wie beim HRTF-Kernel). Neuer Instanz-Parameter `reverbMix` (0..1, wie
  stark das Objekt insgesamt in den Hall einspeist). Send-Betrag wiederverwendet den bereits
  vorhandenen `distAmp` (`1 - distAmp`) — nah (distAmp≈1) sendet kaum was, weit (distAmp≈0)
  sendet fast alles, skaliert mit `reverbMix`.
* Demo-Skripte (`insect_demo.scd`, `sample_demo.scd`, `sample_demo_moving.scd`): `RoomReverb`
  vor den `addSynthDef`-Aufrufen anlegen, Bus durchreichen, `RoomReverb.stop` beim Aufräumen.
  Reihenfolge im Node-Baum bewusst vereinfacht (Reverb-Synth wird zuletzt gestartet, nachdem
  alle anderen Synths schon laufen) — bei später live dazu registrierten Objekten könnte es
  einen Block Verzögerung geben, akzeptiert als Einschränkung der ersten Version.

## Challenges & Solutions

`GVerb`-Signatur vor dem Implementieren per Core-SC-Doku verifiziert (nach den API-Stolperern
in Intent 5/20 lieber nachschauen als raten): `GVerb.ar(in, roomsize, revtime, damping,
inputbw, spread, drylevel, earlyreflevel, taillevel, maxroomsize, mul, add)` — `drylevel` etc.
sind lineare Amplituden, nicht dB (das Beispiel in der Hilfe wandelt explizit per `.dbamp` um).
`drylevel=0` fest verdrahtet, da der Direktschall schon über die Binauralizer läuft.

Beim headless Verifizieren von `AtkBinauralizer.addSynthDef` mit einem unrealistischen
Testfall (Kernel nie via `*setup` gesetzt, also `decoder=nil`) blieb der sclang-Prozess nach
einem `DoesNotUnderstandError` einfach hängen (0% CPU, kein Exit) — der Fehler brach das
restliche Skript ab, bevor die eigene `0.exit`-Zeile erreicht wurde, und ohne die hätte der
Prozess nie von selbst beendet. Kein Bug in `AtkBinauralizer` selbst, nur ein zu
unrealistischer Testfall (im echten Betrieb existiert `decoder` immer, da `*addSynthDef` nur
über `*setup` aufgerufen wird). Für den Rest der Verifikation auf `Binauralizer` beschränkt
(braucht keinen Server) und den bereits erfolgreichen Klassen-Compile von `AtkBinauralizer`
als ausreichend akzeptiert — der echte reverbBus-Zweig braucht ohnehin den Server (Hörtest).

SynthDef-Build zusätzlich headless per `sclang -l conf.yaml script.scd` verifiziert (siehe
Memory-Eintrag zum headless Testlauf) — baut fehlerfrei, bevor der Nutzer testen musste.
