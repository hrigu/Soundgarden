# 15 - Feature (Spatial-Audio): Steady Soundobjekte

## 1. Why

### 1.1 Objective

Soundobjekte unterstützen, die sich nicht von selbst bewegen, deren Position sich aber live
per Code verändern lässt.

### 1.2 Context

Bisher ist jedes `SoundObject` um ein `Movable` mit einer aktiv bewegenden `MoveRule`
aufgebaut. Manchmal ist aber eine stationäre Klangquelle gewünscht (z.B. ein "Brunnen" oder
eine feste Lichtquelle im Raum) — keine Eigenbewegung, aber trotzdem live umplatzierbar
während eines Sets.

### 1.3 Depends On Intents

* 3 - Feature (Spatial-Audio): Binaurales Insekt-Prototyp — liefert `Movable`.
* 9 - Enhancement (Spatial-Audio): Insekt-Architektur aufteilen — liefert die `MoveRule`-Architektur.

## 2.0 What

```gherkin
Feature: Steady Soundobjekte

  Als Nutzer, der live codet
  möchte ich ein Soundobjekt anlegen können, das von selbst nicht wandert
  damit ich stationäre Klangquellen platzieren kann, deren Position ich bei Bedarf trotzdem
  live verändere.

  Scenario: SteadyMoveRule hält die Position
    Given eine SteadyMoveRule und eine Startposition
    When next(pos, t, dt) für beliebige t/dt aufgerufen wird
    Then bleibt die zurückgegebene Position identisch zur übergebenen pos

  Scenario: Position live verändern
    Given ein Movable mit SteadyMoveRule
    When moveTo(newPos) aufgerufen wird
    Then ist movable.pos exakt newPos, unabhängig von roomRadius
```

## 3.0 How

### 3.1 Implementation Context

* Neu `classes/sg/soundobjects/SteadyMoveRule.sc` — `MoveRule`-Subklasse, `next` gibt `pos`
  unverändert zurück (ignoriert `t`/`dt`).
* `classes/sg/soundobjects/Movable.sc` — neue Methode `moveTo(newPos)`: setzt `pos` direkt,
  **ohne** die `roomRadius`-Rückführung aus `step()` — die ist für organische Bewegungsdrift
  gedacht, nicht für bewusste Live-Coding-Platzierung.
* Tests: `tests/sg/soundobjects/TestSteadyMoveRule.sc` + Testfall für `moveTo` (reine Logik,
  kein Server nötig).
