# 31 - Research (Spatial-Audio): Puls/Chirp-Zeitmodell für Zirp-Pattern aus Bioakustik-Quellen

## 1. Why

### 1.1 Objective
Bevor ein Pattern-System für `InsectSound` gebaut wird (siehe Diskussion zu einem möglichen
Folge-Intent "Zirp-Pattern, Phase 1"), ein aus echten Quellen abgeleitetes, quantitatives
Timing-Modell der Grillen-Zirp-Rhythmik festhalten — statt frei erfundener Platzhalter-Werte
für Dauer/Pause. Ergebnis ist ein dokumentiertes Zahlenmodell plus die Entscheidung, ob eine
flache Segmentliste (`[dauer, istAn]`-Paare) genügt oder eine verschachtelte Struktur (Pattern
von Patterns) nötig ist.

### 1.2 Context
Zwei vom Nutzer benannte Quellen, im Volltext ausgewertet:

* songsofinsects.com, "Biology of Insect Song" — begriffliche Grundlage: *Chirp* (kurzer
  Notenstoß in Serie) vs. *Trill* (kontinuierliches, zu schnelles Band von Pulsen); manche
  Arten singen "mehr oder weniger kontinuierlich", andere mit langen Stille-Pausen zwischen
  Gesangssequenzen — zwei grundverschiedene Rhythmus-Familien, nicht nur eine. Pausen zwischen
  Pulsen sind artspezifisch und erkennungskritisch.
* PMC7603463 (Järvilehto et al., *Gryllus bimaculatus*) — quantitatives Modell:

  | Ebene | Wert |
  |---|---|
  | Chirprate | ~3 Hz → Chirp-Periode ~300–500 ms |
  | Chirpdauer | ~160–171 ms (unter neuronaler Stimulation 230–400+ ms) |
  | Pulse pro Chirp | 3–4 |
  | Pulsrate (angegeben) | ~30 Hz |
  | Interpuls-Intervall (gemessen) | ~50 ms |
  | Steuerung | A3–A4-Ganglien: Puls-Timer (30 Hz); A4–A6: Chirp-Timer (3 Hz), Aktivierungskaskade posterior→anterior |

  **Offene Unstimmigkeit (bewusst nicht glattgebügelt):** 30 Hz Pulsrate ⇒ ~33 ms Periode,
  aber das gemessene Interpuls-Intervall ist ~50 ms (≈20 Hz). Vermutung: "30 Hz" bezeichnet die
  interne Oszillator-/Trägerfrequenz der Flügelbewegung, die 50 ms den tatsächlichen Abstand
  hörbarer Pulse — beide Quellen klären das nicht. Für Task 2.0 wird das Interpuls-Intervall
  (50 ms, direkt gemessen) als belastbarer angesehen als die Pulsrate (nicht näher
  spezifiziert, ob Träger- oder Ereignisfrequenz) — siehe Task 2.0/3.0.

## 2.0 What

Ergebnis dieses Research-Intents ist ein dokumentiertes, aus den beiden Quellen abgeleitetes
Zwei-Ebenen-Zeitmodell (Puls innerhalb Chirp, Chirp innerhalb Pause-Zyklus) mit konkreten
Zahlenwerten/Wertebereichen, plus:

* die geprüfte Aussage, ob sich dieses Zwei-Ebenen-Modell verlustfrei in eine flache
  `[dauer, istAn]`-Segmentliste auflösen lässt (Ergebnis siehe Task 2.0/3.0: ja — ein Chirp aus
  N Pulsen wird zu `[[pulseOn, true], [pulseOff, false]]` N-mal wiederholt, gefolgt von einem
  Segment `[interchirpPause, false]`);
* einen konkreten Default-Zahlensatz (sclang-Literal), den ein künftiger Feature-Intent direkt
  übernehmen kann, statt selbst Zahlen zu erfinden;
* die offene Unstimmigkeit (30 Hz vs. 50 ms) explizit vermerkt, nicht aufgelöst.

Keine Gherkin-Szenarien — reines Recherche-Ergebnis, keine Implementierung, kein Hörtest
(siehe AGENTS.md: Research-Intents haben eine kurze Ergebnisbeschreibung statt Gherkin).

## Challenges & Solutions

### Task 2.0 — flache Segmentliste reicht, keine Verschachtelung nötig
Das Zwei-Ebenen-Modell (Puls innerhalb Chirp, Chirp innerhalb eines Interchirp-Zyklus) lässt
sich verlustfrei in eine flache `[dauer, istAn]`-Liste **ausschreiben**: Wiederholung auf
Puls-Ebene (N Pulse pro Chirp) wird zu N literal wiederholten `[on, off]`-Paaren, Wiederholung
auf Chirp-Ebene übernimmt `CallingPattern`s ohnehin vorhandenes zyklisches Wrap-Around
(`next` springt nach dem letzten Segment zurück auf Index 0) — die gesamte Chirp+Pause-Periode
ist einfach *ein* Durchlauf durch die Liste, der sich endlos wiederholt. Eine verschachtelte
Pattern-von-Patterns-Struktur würde hier keine zusätzliche Ausdruckskraft bringen, nur mehr
Code — bestätigt den in 1.2/3.1 (Intent-Entwurf für die spätere Implementierung) skizzierten
Ansatz. Verschachtelung würde erst relevant, falls Puls- und Chirp-Ebene *unabhängig*
variieren müssten (z.B. Puls-Jitter bei sonst fixer Chirp-Struktur) — das ist Phase-2-Scope
(Unregelmäßigkeit), nicht Teil dieses Research-Intents.

### Task 3.0 — abgeleiteter Default-Zahlensatz
Sourced-Werte allein legen die Aufteilung *innerhalb* eines 50-ms-Interpuls-Intervalls in
"hörbar"/"still" nicht fest — keine der beiden Quellen nennt einen Pulsdauer/Pause-Duty-Cycle.
Angenommen (nicht sourced, hier explizit als Annahme markiert): ~40% hörbar / 60% still pro
Puls-Intervall, angelehnt an typische kurze Stridulations-Bursts:

```supercollider
// Eine Chirp+Pause-Periode, generiert aus 4 Pulsen × 50ms + Interchirp-Pause 200ms
// (sourced: 3–4 Pulse/Chirp, ~50ms Interpuls-Intervall, ~300–500ms Chirp-Periode)
~cricketPattern = CallingPattern.new([
	[0.02, true], [0.03, false],  // Puls 1 (an ~40%, aus ~60% von 50ms)
	[0.02, true], [0.03, false],  // Puls 2
	[0.02, true], [0.03, false],  // Puls 3
	[0.02, true], [0.03, false],  // Puls 4
	[0.2,  false]                 // Interchirp-Pause
]);
// Chirpdauer (4×50ms) = 200ms — nahe am sourced-Bereich 160–171ms (normal) bzw.
// 230–400ms+ (stimuliert); Gesamtperiode 400ms ⇒ ~2.5 Hz, innerhalb des sourced
// 300–500ms-Periodenbereichs (~2–3.3 Hz).
```

Alle vier Zahlen (Pulsanzahl 4, Interpuls-Intervall 50ms, resultierende Chirp-/Periodendauer)
sind direkt aus 1.2 abgeleitet und liegen innerhalb der sourced-Bereiche; nur der interne
40/60-Duty-Cycle pro Puls ist eine unsourcte Annahme, die ein künftiger Feature-Intent bei
Bedarf per Hörtest nachjustieren kann (siehe AGENTS.md: Klangverhalten wird per Ohr verifiziert,
nicht vorab bewiesen).
