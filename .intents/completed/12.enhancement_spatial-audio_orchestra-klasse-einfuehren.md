# 12 - Enhancement (Spatial-Audio): Orchestra-Klasse einführen, SoundObject von Routine und Listener entkoppeln

## 1. Why

### 1.1 Objective

Jedes `SoundObject` startet aktuell seine eigene `Routine`, kennt den `Listener` direkt und
schiebt Azimuth/Distanz selbst in seinen `Binauralizer`. Das koppelt `SoundObject` unnötig an
`Listener` und dupliziert den Tick-Mechanismus pro Objekt. Eine zentrale `Orchestra`-Klasse hält
den einen `Listener`, tickt alle registrierten `SoundObject`s mit einem gemeinsamen Taktgeber und
berechnet Azimuth/Distanz zentral — `SoundObject` muss den `Listener` dann gar nicht mehr kennen.

### 1.2 Context

Ergebnis der Diskussion nach Intent 10/11: der Nutzer möchte weg von "jedes SoundObject hat seine
eigene Routine + kennt den Listener" hin zu einem zentralen "Orchester", das alle Klangobjekte
periodisch anstösst. `Orchestra.stop` übernimmt dabei auch das Stoppen aller registrierten
SoundObjects — ein Aufruf stellt die ganze Szene still, genauso wie bisher `~insect.stop` das für
ein einzelnes Insekt tat.

## 2.0 What

```gherkin
Feature: Orchestra tickt und verwaltet alle SoundObjects zentral

  Als Entwickler, der mehrere Klangobjekte gleichzeitig orchestriert
  möchte ich einen zentralen Taktgeber statt einer Routine pro Objekt
  damit SoundObject den Listener nicht mehr kennen muss und die ganze Szene über einen
  Aufruf gestartet/gestoppt werden kann.

  Scenario: SoundObject kennt den Listener nicht mehr
    Given ein SoundObject ohne Listener-Referenz im Konstruktor
    When es bei einer Orchestra-Instanz registriert wird
    Then läuft die Binauralisierung trotzdem korrekt

  Scenario: Orchestra tickt alle registrierten SoundObjects zentral
    Given eine Orchestra-Instanz mit Listener und mehreren registrierten SoundObjects
    When orchestra.play aufgerufen wird
    Then bewegt sich jedes registrierte SoundObject weiter
    And erhält jedes registrierte SoundObject seine aktuelle Azimuth/Distanz über updateSpatial

  Scenario: Orchestra stoppt die ganze Szene über einen Aufruf
    Given eine Orchestra-Instanz mit mehreren registrierten, spielenden SoundObjects
    When orchestra.stop aufgerufen wird
    Then stoppt das Ticken
    And wird stop auf jedem registrierten SoundObject aufgerufen (Synths/Busse freigegeben)

  Scenario: Tick-Logik ist ohne Server testbar
    Given eine Orchestra-Instanz mit Fake-Listener und Fake-SoundObjects (Doubles mit
      pos/step/updateSpatial)
    When ein Tick ausgeführt wird
    Then ruft Orchestra step() und updateSpatial() mit den erwarteten Werten auf jedem Fake auf

  Scenario: Bestehendes Hörverhalten bleibt identisch
    Given insect_demo.scd, umgestellt auf Orchestra statt Einzel-Routinen
    When beide Insekten abgespielt werden
    Then klingen sie wie vorher (Hörtest)
```

## 3.0 How

### 3.1 Implementation Context

* Neu `classes/Orchestra.sc` — hält `listener` und eine Liste registrierter `SoundObject`s
  (`register`/`unregister`). `play(server, updateRate = 30)` ruft zuerst `soundObject.play(server)`
  auf jedem registrierten Objekt auf (Synth-Start) und startet danach eine **einzige** Routine,
  die pro Tick für jedes registrierte Objekt `so.step(dt)` aufruft, Azimuth/Distanz via
  `listener.relativeAzimuth(so.pos)`/`distanceTo(so.pos)` berechnet und
  `so.updateSpatial(az, dist)` aufruft. `stop` beendet die Routine, ruft `stop` auf jedem
  registrierten Objekt auf und leert die Registry. Damit reicht ein Aufruf
  (`orchestra.play(s)`/`orchestra.stop`), statt jedes SoundObject einzeln zu starten/stoppen.
  *(Revidiert nach erstem Hörtest: ursprünglich sollte `play` nur ticken, `SoundObject.play`
  separat aufgerufen werden — auf Wunsch des Nutzers zusammengelegt, da das separate Starten
  unschön war.)*
* `classes/SoundObject.sc` — Konstruktor verliert den `listener`-Parameter; eigene `Routine`
  entfällt komplett (kein `routine`-Var, kein `play`-Loop mehr dafür). Neue schlanke Methoden:
  `pos` (delegiert an `movable.pos`), `step(dt)` (delegiert an `movable.step(dt)`),
  `updateSpatial(az, dist)` (delegiert an `binauralizer.set(az, dist)`). `play(server)`/`stop`
  bleiben fürs Sound-/Binauralizer-Wiring zuständig (Synth-Start/-Stop), werden aber von
  `Orchestra.play`/`stop` aufgerufen statt einzeln von aussen.
* `insect_demo.scd` — eine `Orchestra`-Instanz mit dem gemeinsamen `Listener` anlegen, beide
  Insekten registrieren, `orchestra.play(s)`/`orchestra.stop` statt Einzel-Routinen und
  Einzel-`play`-Aufrufen.
* Neu `tests/TestOrchestra.sc` — UnitTest für die Tick- und Stop-Logik mit Fake-`Listener`/
  Fake-`SoundObject`s (reine sclang-Logik, kein Server nötig — analog `TestCircularMoveRule`).
* `README.md`, `docs/insekt-architektur-walkthrough.md` entsprechend anpassen.

## 4. Challenges & Solutions

* Beim Hörtest (Task 5.0) fiel auf: `~insect1.play(s); ~insect2.play(s); ~orchestra.play;` in
  `insect_demo.scd` erforderte, jedes SoundObject einzeln zu starten und danach separat das
  Orchester — unschön, weil es zwei verschiedene Startvorgänge für eine Szene sind. Gelöst durch
  Erweiterung von `Orchestra.play` um den `server`-Parameter: es ruft jetzt zuerst
  `soundObject.play(server)` auf jedem registrierten Objekt auf und startet danach die
  Tick-Routine — ein einziger Aufruf (`orchestra.play(s)`) für die ganze Szene, symmetrisch zu
  `orchestra.stop`, das schon vorher alle registrierten Objekte mitstoppte.
* Dabei zusätzlich einen Dokumentationsfehler in `insect_demo.scd` korrigiert: Der Kommentar
  listete `boot.scd` vor `load_classes.scd`, tatsächlich muss `load_classes.scd` zuerst laufen
  (siehe `AGENTS.md`/`CLAUDE.md`). Als eigener, trivialer Bug-Commit (kein Intent-Task) behandelt.
