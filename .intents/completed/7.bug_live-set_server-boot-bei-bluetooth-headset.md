# 7 - Bug (Live-Set): Server-Boot schlägt bei angeschlossenem Bluetooth-Headset fehl

## 1. Why

### 1.1 Objective

`boot.scd` soll zuverlässig booten, unabhängig davon, welche Audiogeräte gerade angeschlossen
sind — insbesondere Bluetooth-Kopfhörer.

### 1.2 Context

Mit angeschlossenem Bose AE2 SoundLink (Bluetooth) lieferte das System dessen Mikrofon-Profil
mit nur 16kHz Samplerate, während der Output mit 44100Hz lief:

```
WARNING: Input sample rate is 16000, but output is 44100. Attempting to set input sample rate
to match the output.
ERROR: Setting sample rate failed. OSStatus epon
could not initialize audio.
Server 'localhost' exited with exit code 0.
```

Das Projekt nutzt an keiner Stelle Audio-Input (Mikrofon) — nur Wiedergabe (`PlayBuf`) und
Synthese. Der Input-Kanal muss also gar nicht erst aktiv sein.

## 2.0 What

```gherkin
Feature: Server-Boot unabhängig von angeschlossenen Audiogeräten

  Als Nutzer
  möchte ich, dass boot.scd zuverlässig bootet
  damit ein zufällig verbundenes Bluetooth-Headset (oder ein anderes Gerät mit abweichender
  Input-Samplerate) den Start nicht blockiert.

  Scenario: Bluetooth-Headset mit abweichender Mikrofon-Samplerate ist verbunden
    Given ein Bluetooth-Headset mit 16kHz-Mikrofon ist verbunden, Output läuft mit 44100Hz
    When boot.scd ausgeführt wird
    Then bootet der Server erfolgreich, ohne Samplerate-Konflikt
```

## 3.0 How

### 3.1 Implementation Context

`s.options.numInputBusChannels = 0` deaktiviert den Audio-Input komplett, bevor `s.waitForBoot`
läuft — dadurch muss keine Input-Samplerate mehr mit der Output-Samplerate abgeglichen werden,
unabhängig vom angeschlossenen Gerät.

### 3.2 Tasks

* [x] 1.0 - Fix
    - [x] 1.1 - `boot.scd`: `s.options.numInputBusChannels = 0` setzen

## 4. Challenges & Solutions

`s.options.numInputBusChannels = 0` allein reichte nicht für sauberen Stereo-Klang über das
Bluetooth-Headset: Solange **macOS selbst** (Systemeinstellungen → Ton → Eingabe) das Headset
noch als Standard-Mikrofon führte, hielt das System die Bluetooth-Verbindung systemweit im
HFP-Modus (Freisprechen, mono, komprimierter Codec) — hörbar an Rauschen, Verzerrung und
fehlendem Stereo-Panning, unabhängig davon, was SuperCollider anfragt. Gelöst durch Umstellen
des Standard-Eingabegeräts in den macOS-Systemeinstellungen auf ein Nicht-Bluetooth-Gerät (z.B.
das eingebaute Mikrofon) — danach handelt macOS die Verbindung neu aus und nutzt A2DP
(hochwertige Stereo-Wiedergabe).
