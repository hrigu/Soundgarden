# 2 - Test (Boot-Tracks): Automatisierte Tests für Track-Extension-Erkennung

## 1. Why

### 1.1 Objective

Die Extension-Erkennungslogik aus `boot.scd` automatisiert absichern, ohne Server oder
echte Audiodateien zu brauchen.

### 1.2 Context

Genau diese Logik war zuvor durch den Symbol/String-Vergleichsbug lahmgelegt (Intent 1,
Challenges & Solutions). Ein Test hätte das sofort sichtbar gemacht, statt es erst beim
Live-Debugging im Post-Fenster zu finden.

### 1.3 Depends On Intents

* 1 - Feature (Live-Set): Live-Coding-Rig Grundgerüst — enthält die zu testende Logik.

## 2.0 What

```gherkin
Feature: Track-Extension-Erkennung

  Als Entwickler
  möchte ich die Extension-Filterlogik automatisiert testen
  damit ein Regressions-Bug wie der Symbol/String-Vergleichsfehler sofort auffällt.

  Scenario: Symbol-Extensions werden erkannt
    Given eine Extension als Symbol, z.B. \aiff
    When BootTrackDetection.isValidExtension aufgerufen wird
    Then wird sie als gültig erkannt

  Scenario: String-Extensions werden erkannt
    Given eine Extension als String, z.B. "aiff"
    When BootTrackDetection.isValidExtension aufgerufen wird
    Then wird sie als gültig erkannt

  Scenario: Groß-/Kleinschreibung spielt keine Rolle
    Given die Extension "AIFF"
    When BootTrackDetection.isValidExtension aufgerufen wird
    Then wird sie als gültig erkannt

  Scenario: Nicht unterstützte Extensions werden abgelehnt
    Given die Extension "mp3"
    When BootTrackDetection.isValidExtension aufgerufen wird
    Then wird sie abgelehnt
```

## 3.0 How

### 3.1 Implementation Context

* `classes/BootTrackDetection.sc` — die Extension-Logik als eigene Klasse, damit sie ohne
  Server testbar ist; `boot.scd` ruft dieselbe Klasse auf (keine doppelte Logik).
* `classes/TestBootTrackDetection.sc` — `UnitTest`-Subklasse mit vier Testfällen.
* `load_classes.scd` bindet `classes/` in den Klassenpfad ein (`LanguageConfig.addIncludePath`
  + `thisProcess.recompile`).
* `run_tests.scd` führt die Suite aus.
