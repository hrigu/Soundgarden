# 8 - Bug (Boot): Doku nennt falsche Reihenfolge für load_classes.scd

## 1. Why

### 1.1 Objective

Die dokumentierte Ausführungsreihenfolge korrigieren: `load_classes.scd` muss **vor**
`boot.scd` laufen, nicht danach.

### 1.2 Context

Seit Intent 2 ruft `boot.scd` `BootTrackDetection.isValidExtension(...)` auf. Diese Klasse
existiert erst, nachdem `load_classes.scd` den `classes/`-Ordner in den Klassenpfad
eingebunden und die Klassenbibliothek neu kompiliert hat. README.md und CLAUDE.md nannten aber
weiterhin `boot.scd` als ersten Schritt (korrekte Reihenfolge nur für den Spatial-Audio-Workflow
zufällig anders herum dokumentiert) — bei einem frischen SuperCollider-Start (Klassenbibliothek
im Auslieferungszustand) führt `boot.scd` zuerst zu `ERROR: Class not defined.`

### 1.3 Depends On Intents

* 2 - Test (Boot-Tracks): Automatisierte Tests für Track-Extension-Erkennung — führte die
  Abhängigkeit von `BootTrackDetection` in `boot.scd` ein.

## 2.0 What

```gherkin
    When ich load_classes.scd und danach boot.scd ausführe (wie neu dokumentiert)
    Then bootet der Server ohne "Class not defined"-Fehler
```

### 3.1 Implementation Context

Reine Doku-Korrektur, kein Code-Fix nötig — die Abhängigkeit selbst ist gewollt (siehe Intent
2), nur die dokumentierte Reihenfolge war falsch/inkonsistent.
