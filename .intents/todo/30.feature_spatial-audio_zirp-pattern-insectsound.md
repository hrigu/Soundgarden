# 30 - Feature (Spatial-Audio): Zirp-Pattern für InsectSound (Phase 1: festes An/Aus-Muster)

## 1. Why

### 1.1 Objective
`InsectSound` soll nicht mehr zwingend durchgehend klingen, sondern einem einfachen, fest
vorgegebenen An/Aus-Rhythmus folgen können (z.B. "kurzer Zirp, lange Pause, kurzer Zirp, lange
Pause") — über einen neuen, optionalen Parameter, nicht über Skript-Logik in den Demos. Ohne
diesen Parameter bleibt das bisherige, durchgehende Verhalten unverändert (keine Bruch-Änderung
für `demos/insects.scd`).

Bewusst nur Phase 1 (festes Muster) — unregelmäßige Muster (Zufallsstreuung pro Segment) und
die Anbindung an SC-eigene Patterns (`Pseq`/`Pwhite`/`Rest`) sind eigene, spätere Intents, siehe
1.3.

### 1.2 Context
Intent 31 (Research, nachträglich mit höherer Nummer angelegt, siehe dessen 1.3) liefert das
fachliche Zahlenmodell: reale Grillen-Zirp-Rhythmik ist zweistufig (Puls innerhalb Chirp, Chirp
innerhalb einer Interchirp-Pause), lässt sich aber verlustfrei in eine flache
`[dauer, istAn]`-Segmentliste ausschreiben (kein Pattern-von-Patterns nötig). Intent 31 liefert
auch einen konkreten, quellenbasierten Default-Zahlensatz (4 Pulse à 50ms + 200ms
Interchirp-Pause), der hier direkt für Task 4.0 (Demo) übernommen wird.

SuperCollider bringt mit `Pbind`/`Pseq`/`Rest` bereits eine Pattern-Sprache für genau solche
Rhythmen mit — bewusst NICHT in Phase 1 verwendet: reine Literal-Daten (`[dauer, istAn]`-Paare)
sind ohne Server testbar (passt zum bestehenden TDD-Ansatz für reine sclang-Logik, siehe
CLAUDE.md). Spätere Phasen (Zufallsstreuung, volle SC-Pattern-Anbindung) bauen auf demselben
`next`-Interface auf, ohne Phase 1 zu brechen.

`InsectSound`s Signal läuft aktuell durchgehend mit konstantem `amp` (keine Möglichkeit,
zwischendurch stumm zu sein) — `SoundObjectBuilder`/Room-Wiring (Intent 29) sind davon nicht
betroffen: `pattern` wird einfach ein weiterer Key im `soundParams`-Event, keine Änderung an
`SoundObjectBuilder` nötig.

### 1.3 Related To Intents

* 29 - SoundObjectBuilder — `pattern` ist nur ein zusätzlicher `soundParams`-Key, komponiert
  ohne Anpassung.
* 31 - Research: Puls/Chirp-Zeitmodell — liefert das Zahlenmodell und die
  Flach-Segmentliste-Entscheidung, auf der dieser Intent aufbaut (siehe 1.2).
* Geplante Folge-Intents (noch nicht angelegt): Phase 2 (unregelmäßige Muster per
  Zufallsbereich statt fixer Dauer), Phase 3 (Pattern-Player akzeptiert direkt SC-Patterns via
  Duck-Typing `next`/`asStream`, für volle Ausdruckskraft ohne weiteren Code hier).

## 2.0 What

```gherkin
Feature: Zirp-Pattern für InsectSound (festes An/Aus-Muster)

  Als Person, die ein Insekt konfiguriert
  möchte ich ein einfaches, festes An/Aus-Muster angeben können
  damit es nicht mehr zwingend durchgehend zirpt.

  Scenario: CallingPattern liefert Segmente zyklisch
    Given ein CallingPattern mit Segmenten [[0.15, true], [0.05, false]]
    When ich next dreimal aufrufe
    Then erhalte ich [0.15, true], [0.05, false], [0.15, true] (zurück zum Anfang)

  Scenario: InsectSound ohne pattern-Parameter klingt wie bisher durchgehend
    Given eine InsectSound-Instanz ohne pattern
    When sie über den Server abgespielt wird (Hörtest)
    Then läuft sie unverändert durchgehend wie vor diesem Intent

  Scenario: InsectSound mit pattern gated An/Aus
    Given eine InsectSound-Instanz mit einem CallingPattern (kurz an, lang aus)
    When sie abgespielt wird (Hörtest)
    Then ist der Klang nur während der "an"-Segmente hörbar, während "aus" still — ohne
      hörbaren Klick beim Umschalten
```

## 3.0 How

### 3.1 Implementation Context

* Neue Klasse `classes/sg/sounds/CallingPattern.sc` — reine sclang-Logik, kein Server nötig:
  - `var <segments` (Array von `[dauer, istAn]`-Paaren), `var <>index` (aktuelle Position,
    Default 0).
  - `*new { |segments| ... }`.
  - `next` — liefert `segments[index]`, erhöht `index`, wrappt bei Erreichen des Endes zurück
    auf 0 (`(index + 1) % segments.size`). Eigenes `next`-Interface statt SC-`Stream`-Protokoll
    — bewusst minimal, siehe 1.2 (spätere Phase 3 kann stattdessen `Pseq(...).asStream` an
    dieselbe Aufrufstelle in `InsectSound` übergeben, sobald diese nur `next` erwartet — kein
    API-Bruch für Phase 1).
* `classes/sg/sounds/InsectSound.sc`:
  - Neue Instanzvariable `var <>pattern` (Default nil = bisheriges, durchgehendes Verhalten),
    `var patternRoutine` (intern).
  - `*new`/`init` um `pattern`-Parameter erweitern (letzter Keyword-Arg, Default nil).
  - `play { |server| ... }` überschreiben: `super.play(server)` (Bus+Synth wie bisher über
    `Sound>>play`/`makeSynth`), danach falls `pattern.notNil`: `patternRoutine = Routine({
    loop { var segment = pattern.next; synth.set(\gate, segment[1].asInteger);
    segment[0].wait } }).play`.
  - `stop { patternRoutine.stop; super.stop }` — Routine vor dem Synth-Free stoppen (sonst
    `.set` auf einen bereits freigegebenen Synth, harmlos aber unnötig).
  - `*addSynthDef`: neuer SynthDef-Control `gate = 1` (Default an, wie bisher durchgehend);
    Signal vor `Out.ar` mit `Lag.kr(gate, 0.02)` multiplizieren — sanfte Rampe statt hartem
    Sprung, verhindert Klicks beim Umschalten (siehe Gherkin-Szenario 3). `makeSynth` selbst
    unverändert (übergibt `gate` nicht explizit — dynamischer Laufzeit-Control, analog
    `t_call`, das ebenfalls nicht in `makeSynth`s Args steht).
* `tests/sg/sounds/TestCallingPattern.sc` (neuer Ordner, spiegelt `classes/sg/sounds/` — erste
  reine sclang-Logik-Klasse unter `sounds/`): testet zyklisches `next`, Wrap-Around bei
  verschiedenen Segmentzahlen (1, 2, mehr).
* `demos/insects.scd`: mind. ein Insekt (z.B. `~insect2` oder eine der 10 Varianten) bekommt den
  Cricket-Pattern-Zahlensatz aus Intent 31 (4 Pulse à 50ms, 200ms Interchirp-Pause) als
  `pattern`, um den Hörtest direkt zu ermöglichen.
