# 53 - Feature (Spatial-Audio): SampleSound spielt einen Ausschnitt fester Länge

## 1. Why

### 1.1 Objective
`SampleSound` um einen konfigurierbaren Ausschnitt erweitern (Startpunkt + feste Dauer), damit
Demo-Skripte wie `sample.scd` mehrere, unterschiedlich lange Audiodateien mit *identischer*
hörbarer Länge abspielen können, statt dass jedes Objekt so lange klingt wie die jeweilige
Originaldatei.

### 1.2 Context
`SampleSound.addSynthDef` spielt aktuell über `PlayBuf.ar` immer den kompletten Buffer-Inhalt
ab dessen Anfang ab (kein `startPos`, kein Envelope/Cutoff) — bei `sample.scd` (viele
Soundobjekte im Kreis, alle mit derselben Trigger-Rate) fällt das störend auf, weil kurze
Percussion-Hits und lange Instrumentalphrasen nebeneinander laufen und dadurch hörbar
unterschiedlich lang klingen.

### 1.3 Related To Intents
* 20 - `SampleSound` eingeführt (rhythmisches Sample-Playback).
* 43 - `editableParams`/Live-GUI-Bearbeitung für Sound-Parameter — neue Parameter sollen sich
  konsistent dort einreihen.

## 2.0 What

```gherkin
Feature: SampleSound spielt einen Ausschnitt fester Länge

  Scenario: Standardverhalten bleibt kompatibel
    Given ein SampleSound ohne startFrac/duration
    When ich es abspiele
    Then verhält es sich wie bisher (kompletter Sample-Inhalt, kein Envelope-Cutoff)

  Scenario: Ausschnitt mit fester Dauer
    Given ein SampleSound mit startFrac = 0.1 und duration = 0.5
    When ich es abspiele
    Then beginnt die Wiedergabe bei 10% der Sample-Länge
    And endet nach 0.5s mit einem kurzen Fade-out statt einem hörbaren Klick

  Scenario: mehrere Soundobjekte mit unterschiedlich langen Samples, gleiche duration
    Given zwei SampleSounds aus unterschiedlich langen Dateien, beide mit duration = 0.4
    When ich beide abspiele
    Then klingen beide exakt 0.4s lang, unabhängig von der jeweiligen Dateilänge
```

## 3.0 How

### 3.1 Implementation Context
* `classes/sg/sounds/SampleSound.sc`:
  - neue Instanzvariablen `startFrac` (Default 0, 0..1 — Anteil der Sample-Länge, analog zum
    bereits vorhandenen `phase`-Stil) und `duration` (Default `nil` = altes Verhalten, sonst
    Sekunden).
  - `*addSynthDef`: `PlayBuf.ar`s `startPos`-Argument = `startFrac * BufFrames.kr(buf)`. Wenn
    `duration` gesetzt ist (`durationGate`-Kontrollargument > 0 statt echtem `nil`-Check im
    SynthDef, da SynthDef-Argumente keine `nil`-Werte kennen — 0 bedeutet "kein Cutoff, altes
    Verhalten"), zusätzliches `EnvGen.kr(Env.linen(attackTime, sustainTime, releaseTime), trig,
    doneAction: 0)` als Multiplikator auf das Signal, mit kurzem Attack/Release (~5-10ms) gegen
    Klicks — sustain so berechnet, dass die Gesamtlänge exakt `duration` ergibt.
  - Instanzvariable statt Argument mit `nil`-Default: Default-Konvertierung `duration ?? 0`
    beim Synth-Start (siehe `makeSynth`), analog zum bestehenden `binauralizerClass ?? {
    Binauralizer }`-Muster in `Listener`.
  - `*editableParams` um `startFrac`/`duration` ergänzen (`ControlSpec`), damit beide auch live
    im GUI bearbeitbar sind (Konsistenz mit Intent 43).
* `tests/sg/sounds/TestSampleSound.sc`: `editableParams`-Tests um die neuen Keys erweitern
  (reine sclang-Logik, testbar wie bisher). Das eigentliche SynthDef-Verhalten (Cutoff-Envelope,
  Klickfreiheit) ist Server-seitig und wird wie bei `InsectSound`/`Binauralizer` nicht
  automatisiert getestet, sondern per Hörtest verifiziert (siehe AGENTS.md, TDD-Abschnitt).
* `demos/sample.scd`: gemeinsame `duration` für alle Soundobjekte setzen (z.B. `duration: 0.8`),
  damit der Demo-Zweck ("alle Objekte gleich lang") sichtbar wird.
* Nicht Teil dieses Intents: Zeitdehnung/-stauchung (Pitch-Shift über `rate`), Loop-Handling,
  Crossfades zwischen Retrigger-Zyklen — reines Zuschneiden auf einen festen Ausschnitt.
