# 29 - Feature (Spatial-Audio): Builder für SoundObject mit klonbarer Konfiguration

## 1. Why

### 1.1 Objective
SoundObjects über einen fluent Builder erzeugen, statt `Movable`/`Sound` von Hand zu instanzieren
und an `SoundObject.new` zu übergeben. Der Builder soll klonbar sein: von einer bestehenden
Konfiguration ausgehend nur einzelne Parameter überschreiben, um Varianten zu erzeugen (z.B.
mehrere Insekten mit ähnlichem Klang, aber unterschiedlicher Lautstärke/Frequenz).

### 1.2 Context
`demos/insects.scd` baut aktuell mehrere Insekten von Hand, jedes mit eigenem `Movable`+
`InsectSound`-Verkabelungsblock — Wiederholung ohne Wiederverwendung der gemeinsamen Basis-
Konfiguration. Intent 28 macht die reinen Konstruktor-Aufrufe lesbarer, löst aber nicht das
eigentliche Problem: eine Basis-Konfiguration definieren und daraus mehrere Varianten ableiten.

`Room>>register` baut den Binauralizer aktuell selbst über `listener.makeBinauralizer(reverbMix)`
und erst dann das `SoundObject` (Intent 27: der Binauralizer-Typ ist eine Eigenschaft der Ohren
des Hörers, nicht der Klangquelle — Skripte kennen ihn nicht). Ein Builder mit eigenem
Binauralizer-Slot würde dieses Prinzip umgehen; der Builder bleibt deshalb auf movable+sound
beschränkt, Room bleibt die einzige Stelle, die den Binauralizer bestimmt.

### 1.3 Related To Intents

* 27 - Binauralizer als Interna des Listeners — Binauralizer-Auswahl bleibt Room/Listener
  vorbehalten, davon weicht dieser Intent bewusst nicht ab.
* 28 - Konstruktor-Aufrufe auf Keyword-Args — macht die rohen Aufrufe lesbarer, dieser Intent
  löst das Wiederholungsproblem bei mehreren Varianten.

## 2.0 What

```gherkin
Feature: Builder für SoundObject mit klonbarer Konfiguration

  Als Person, die mehrere ähnliche Soundobjekte aufbaut
  möchte ich eine Basis-Konfiguration einmal definieren und für Varianten klonen
  damit ich nicht bei jeder Variante alle Parameter erneut angeben muss.

  Scenario: SoundObject über den Builder erzeugen
    Given SoundObjectBuilder.new.sound((wingRate: 280, amp: 0.3)).movable((moveRule: CircularMoveRule.new))
    When ich .build(binauralizer) aufrufe
    Then erhalte ich ein fertiges SoundObject mit einer InsectSound-Instanz (wingRate: 280,
      amp: 0.3, alle anderen Parameter Default), einer Movable-Instanz mit der gegebenen
      MoveRule und dem übergebenen Binauralizer

  Scenario: Klon mit überschriebenen Parametern
    Given ein Builder mit sound((wingRate: 280, amp: 0.3))
    When ich ihn klone und .sound((amp: 0.05)) auf dem Klon aufrufe
    Then hat der Klon wingRate weiterhin 280 (übernommen von der Basis) und amp neu 0.05
    And der ursprüngliche Builder bleibt unverändert (amp weiterhin 0.3)

  Scenario: Zwei aus demselben Builder gebaute SoundObjects sind unabhängig
    Given ein Builder, aus dem zwei SoundObjects gebaut werden (build, dann build erneut)
    When ich die Position des einen Movable verändere (step/moveTo)
    Then bleibt die Position des anderen SoundObject unberührt (kein geteilter Movable/Sound-
      State)

  Scenario: Room bestimmt weiterhin den Binauralizer
    Given ein Room mit registrierter Binauralizer-Klasse (addSynthDef)
    When ich room.registerFromBuilder(builder, reverbMix) aufrufe
    Then baut Room den Binauralizer selbst (wie bei register) und reicht ihn an builder.build
      weiter — der Builder selbst kennt keine Binauralizer-Klasse
```

## 3.0 How

### 3.1 Implementation Context

* Neue Klasse `classes/sg/soundobjects/SoundObjectBuilder.sc`:
  - Zwei Slots, je `(class, params)`: `movableClass`/`movableParams` (Default `Movable`),
    `soundClass`/`soundParams` (Default `InsectSound`). `params` je ein `Event`.
  - `movable { |params, aClass| ... ^this }`, analog `sound` — setzen die Klasse (falls
    angegeben, sonst Default) und **mergen** `params` in die bestehenden Params (`putAll`,
    nicht ersetzen) — das ist der Mechanismus für "nur gewisse Parameter ändern". Aufruf z.B.
    `.sound((wingRate: 280, amp: 0.3))`.
  - `clone` — neuer Builder mit **kopierten** Params-Events (`.copy`) je Slot, damit spätere
    Änderungen am Klon die Basis nicht mutieren.
  - `build { |binauralizer| ... }` — für jeden Slot: `instance = class.new; params.keysValuesDo
    { |k, v| instance.perform((k ++ "_").asSymbol, v) }`; danach `SoundObject.new(movableInst,
    soundInst, binauralizer)`. `binauralizer` optional (nil ⇒ `SoundObject`s bestehendes
    Fallback-Verhalten greift). Setzt voraus, dass die Zielklasse für jedes verwendete Param
    einen `<>`-Setter hat (bei `Movable`/`InsectSound`/`SampleSound` bereits durchgehend der
    Fall). Kein `doesNotUnderstand`-Trick — bewusst die schon vorhandenen `<>`-Setter nutzen,
    keine neue Magie.
  - Kein eigener Binauralizer-Slot (siehe 1.2/1.3) — Binauralizer-Auswahl bleibt bei
    Room/Listener.
* `classes/sg/spatial/Room.sc`:
  - Neue Methode `registerFromBuilder { |builder, reverbMix = 0.3| ... }` — baut den
    Binauralizer wie `register` über `listener.makeBinauralizer(reverbMix)`, ruft
    `builder.build(binauralizer)` statt `SoundObject.new(movable, sound, binauralizer)`.
  - Gemeinsame Logik (`orchestra.register(soundObject); ^soundObject`) aus `register` in eine
    private Hilfsmethode (z.B. `finishRegistering { |soundObject| ... }`) extrahieren, von
    beiden Methoden genutzt. `register` selbst bleibt unverändert in seiner Signatur — bestehende
    Aufrufstellen (`demos/sample.scd`, `demos/sample_moving.scd`, `demos/reverb.scd`,
    `tests/sg/spatial/TestRoom.sc`), die movable+sound direkt bauen, bleiben unangetastet.
* `tests/sg/soundobjects/TestSoundObjectBuilder.sc` — reine sclang-Logik, kein Server nötig:
  testet Merge-Semantik, Klon-Unabhängigkeit, `build`-Ergebnis-Typen und dass zwei `build`-
  Aufrufe unabhängige Instanzen liefern.
* `tests/sg/spatial/TestRoom.sc` — Test für `registerFromBuilder` (Fake-Binauralizer-Erzeugung
  wie bestehend, Builder liefert Fake-Movable/-Sound).
* `demos/insects.scd` — mind. eine Insekt-Variante (Klon + überschriebene Parameter) auf
  `SoundObjectBuilder` + `~room.registerFromBuilder` umstellen, um den eigentlichen
  Anwendungsfall zu zeigen.
