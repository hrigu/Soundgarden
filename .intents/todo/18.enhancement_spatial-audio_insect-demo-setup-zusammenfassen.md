# 18 - Enhancement (Spatial-Audio): insect_demo.scd Setup zu einem Block zusammenfassen

## 1. Why

### 1.1 Objective

Der Start-Workflow für die Spatial-Audio-Demo ist aktuell 4 separate Blocks/Zeilen (SynthDefs
registrieren → Objekte anlegen → Orchester starten → Tastatursteuerung starten), mit impliziter
Wartezeit zwischen SynthDef-Registrierung und Objektaufbau, die man bisher manuell durch
getrenntes Ausführen einhält. Ein Block, der die Wartezeit selbst per `s.sync` handhabt,
reduziert das auf einen Cmd+Enter-Druck.

### 1.2 Context

Die Datei ist über mehrere Intents gewachsen (12: Orchestra, 15: Steady-Soundobjekte/Brunnen,
16: Tastatursteuerung) und der Start-Workflow entsprechend länger geworden. `boot.scd` nutzt
bereits dasselbe Muster (`s.waitForBoot({...})` mit `s.sync` darin) — hier wird es für die Demo
übernommen. `load_classes.scd`/`boot.scd` bleiben bewusst separate, einmalige Sitzungsschritte
(gemeinsame Infrastruktur fürs Live-Set-Rig, siehe frühere Diskussion).

### 1.3 Depends On Intents

* 12 - Enhancement (Spatial-Audio): Orchestra-Klasse einführen
* 15 - Feature (Spatial-Audio): Steady-Soundobjekte
* 16 - Feature (Spatial-Audio): Beweglicher Listener per Tastatur

## 2.0 What

```gherkin
Feature: insect_demo.scd Setup in einem Block

  Als Nutzer, der die Spatial-Audio-Demo pro Sitzung mehrfach startet
  möchte ich Setup und Start mit einem einzigen Block ausführen können
  damit ich nicht mehr manuell zwischen SynthDef-Registrierung, Objektaufbau, Orchester- und
  Tastatursteuerung-Start hin- und herspringen muss.

  Scenario: Ein Block startet die Demo vollständig
    Given Server gebootet, Klassen geladen
    When der zusammengefasste Setup-Block ausgeführt wird
    Then sind SynthDefs registriert, alle Soundobjekte angelegt/registriert, Orchester und
      Tastatursteuerung gestartet — ohne weiteren manuellen Zwischenschritt

  Scenario: Live-Coding-Aktion bleibt separat ausführbar
    Given die Demo läuft
    When der Block "Brunnen umplatzieren" einzeln ausgeführt wird
    Then ändert sich nur die Brunnen-Position, ohne die übrige Demo neu zu starten

  Scenario: Ein Block stoppt alles
    Given die Demo läuft (Orchester + Tastatursteuerung)
    When der zusammengefasste Stop-Block ausgeführt wird
    Then stoppen Orchester (inkl. aller registrierten Soundobjekte) und Tastatursteuerung
```

## 3.0 How

### 3.1 Implementation Context

* `insect_demo.scd` — bisherige Blöcke 1 (SynthDefs registrieren), 2 (Objekte anlegen), 3
  (`orchestra.play`) und 3c (Tastatursteuerung starten) zu **einem** Block zusammenfassen,
  umschlossen von `fork { ... }` — nötig, damit `s.sync` nach der SynthDef-Registrierung
  innerhalb des Blocks korrekt wartet, statt den Hauptthread zu blockieren (gleiches Muster wie
  `s.waitForBoot({...})` in `boot.scd`, das intern ebenfalls forkt).
* Bisheriger Block 3b (Brunnen umplatzieren, `~fountainMovable.moveTo(...)`) bleibt bewusst ein
  separater, einzeln ausführbarer Block — Live-Coding-Aktion während des Sets, kein Setup-Schritt.
* Bisheriger Block 4 (`~orchestra.stop`) und die Tastatursteuerung-Stop-Zeile
  (`~listenerControl.stop`) zu einem Block zusammenfassen.
* Kopfkommentar/Blocknummerierung in `insect_demo.scd` entsprechend anpassen.
