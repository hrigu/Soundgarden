# 54 - Enhancement (Spatial-Audio): GUI-Legacy aufräumen und Doku synchronisieren

## 1. Why

### 1.1 Objective

Toten Code aus einer abgelösten GUI-Iteration entfernen, `CLAUDE.md` mit dem tatsächlichen
Klassenbestand synchronisieren und das bisher nur durch Konvention geteilte
Binauralizer-Interface einmal explizit dokumentieren.

### 1.2 Context

Architektur-Review ergab: `SpaceView`/`KeyboardListenerControl` wurden durch
`SpatialControlPanel` abgelöst, sind aber nirgends mehr referenziert (kein Demo, kein Test) und
duplizieren dabei tote Geometrie-Logik. `CLAUDE.md` nennt weder `classes/sg/gui/` noch `Room`,
`RoomReverb`, `AtkBinauralizer`, `DirectOutBinauralizer`, `SoundPresetLibrary`,
`SampleSoundConfigurator`, `SongSound`, `CallingPattern`/`RhythmPatternCreator` — und führt die
hier gelöschten Klassen noch als aktiv. `Binauralizer`/`AtkBinauralizer`/
`DirectOutBinauralizer` teilen sich `play`/`set`/`stop` nur durch Konvention, ohne
dokumentiertes gemeinsames Interface.

## 2.0 What

```gherkin
Feature: GUI-Legacy aufräumen und Doku synchronisieren

  Als Entwickler an diesem Repo
  möchte ich keinen toten Code und eine aktuelle CLAUDE.md vorfinden
  damit die Architektur-Doku als verlässliche Landkarte funktioniert.

  Scenario: Toter Code entfernt
    Given SpaceView und KeyboardListenerControl werden von keinem Demo/Test referenziert
    When sie gelöscht werden
    Then bleibt run_tests.scd grün und kein Demo bricht

  Scenario: CLAUDE.md aktuell
    Given der tatsächliche Klassenbestand unter classes/sg/
    When CLAUDE.md gelesen wird
    Then sind alle Klassen/Ordner dort auffindbar, keine toten Verweise mehr

  Scenario: Binauralizer-Interface dokumentiert
    Given drei Binauralizer-Implementierungen mit identischem play/set/stop-Vertrag
    When ihre Klassenkommentare gelesen werden
    Then ist der gemeinsame Vertrag an einer kanonischen Stelle explizit benannt
```

## 3.0 How

### 3.1 Implementation Context

Reine Struktur-/Doku-Änderungen, keine SynthDef-/Klanglogik-Änderungen. Keine neue
Abstraktionsklasse für Binauralizer (bei 3 Implementierungen unnötig) — nur Dokumentation der
bestehenden Duck-Typing-Konvention.
